2. Question

Category: CSAA – Design Secure Architectures

A tech company that you are working for has undertaken a Total Cost Of Ownership (TCO) analysis evaluating the use of Amazon S3 versus acquiring more storage hardware. The result was that all 1200 employees would be granted access to use Amazon S3 for the storage of their personal documents.

Which of the following will you need to consider so you can set up a solution that incorporates a single sign-on feature from your corporate AD or LDAP directory and also restricts access for each individual user to a designated user folder in an S3 bucket? (Select TWO.)

- Set up a matching IAM user for each of the 1200 users in your corporate directory that needs access to a folder in the S3 bucket.
- `Set up a Federation proxy or an Identity provider, and use AWS Security Token Service to generate temporary tokens.`
- Use 3rd party Single Sign-On solutions such as Atlassian Crowd, OKTA, OneLogin and many others.
- `Configure an IAM role and an IAM Policy to access the bucket.`
- Map each individual user to a designated user folder in S3 using Amazon WorkDocs to access their personal documents.

This is a classic **IAM Federation + STS** question. The answer becomes easy once you identify the keywords.

---

# Step 1: Identify the keywords

The question says:

* **1200 employees**
* **Corporate AD / LDAP**
* **Single Sign-On (SSO)**
* **Access S3**
* **Each user only accesses their own folder**

Immediately think:

> **Enterprise Federation**

Not IAM Users.

---

# Architecture

```text
Employee
      │
      ▼
Corporate AD / LDAP
      │
      ▼
Identity Provider (IdP) / Federation
      │
      ▼
AWS STS
      │
Temporary Credentials
      │
      ▼
IAM Role
      │
IAM Policy
      │
      ▼
Amazon S3
```

---

# Why not IAM Users?

Imagine creating:

* 1200 IAM Users
* 1200 passwords
* 1200 MFA devices
* 1200 permission policies

This is a management nightmare.

AWS recommends:

> Use your existing corporate directory.

---

# Why STS?

STS stands for:

> **Security Token Service**

It provides:

* Temporary credentials
* Temporary Access Key
* Temporary Secret Key
* Temporary Session Token

Instead of permanent IAM credentials.

Think:

```text
Employee logs in

↓

Corporate AD verifies

↓

STS issues temporary credentials

↓

Access S3

↓

Credentials expire
```

This is much safer.

---

# Why IAM Role?

Users don't receive permissions directly.

They assume an IAM Role.

Example:

```
Employee

↓

Assume Role

↓

S3-PersonalFolderRole

↓

Policy:

Allow access only to

s3://company-bucket/${username}/*
```

The role determines what the user can access.

---

# Correct Answers

### ✅ Setup a Federation proxy or Identity Provider + AWS STS

Because:

* Enables SSO
* Uses existing AD/LDAP
* Issues temporary credentials

---

### ✅ Configure IAM Role + IAM Policy

Because:

* Roles contain permissions
* Policies restrict each user to their own folder

---

# Why not 1200 IAM Users?

❌ Terrible management

AWS best practice:

> Use federation instead.

---

# Why not Okta / OneLogin?

This option is tricky.

These products **can** work.

But the question asks:

> **What do you need?**

AWS already provides:

* Federation
* STS

Third-party tools are optional.

---

# Why not WorkDocs?

WorkDocs is:

* File collaboration
* Document editing

It is **not** used to map users to S3 folders.

---

# SAA Memory Trick

Whenever you see:

* Active Directory
* LDAP
* Corporate users
* Single Sign-On
* Thousands of employees
* Temporary access

Immediately think:

> **Federation → STS → IAM Role**

---

## One-line summary

| Requirement                             | AWS Service                    |
| --------------------------------------- | ------------------------------ |
| Corporate login                         | Identity Provider / Federation |
| Temporary credentials                   | STS                            |
| Permissions                             | IAM Role + IAM Policy          |
| Storage                                 | S3                             |
| **Don't create thousands of IAM users** | Use Federation instead         |

### Exam shortcut

If the question mentions **AD/LDAP + SSO + many employees**, the answer is almost always:

> **Identity Federation + AWS STS + IAM Roles (not IAM Users).**
