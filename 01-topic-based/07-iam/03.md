3. Question

Category: CSAA – Design Secure Architectures

A Solutions Architect is managing a company’s AWS account of approximately 300 IAM users. The company has a new policy that requires changing the associated permissions of all 100 IAM users who control access to Amazon S3 buckets.

What will the Solutions Architect do to avoid the time-consuming task of applying the policy to each user?

- Create a new IAM role and add each user to the IAM role.
- Create a scheduled rule with Amazon EventBridge Scheduler to automatically apply the policy to all 100 IAM users at once.
- `Create a new IAM group and then add the users that require access to the S3 bucket. Afterwards, apply the policy to IAM group.`
- Create a new S3 bucket access policy with unlimited access for each IAM user.

This is one of the easiest but most frequently asked IAM questions in SAA.

---

# Keywords

* **300 IAM users**
* **100 users need same permissions**
* **Easy management**
* **Avoid updating each user**

The keyword is:

> **Same permissions for many users**

Immediately think:

> **IAM Group**

---

# AWS IAM hierarchy

```text
IAM Users
      │
      ▼
IAM Group
      │
      ▼
IAM Policy
      │
      ▼
Permissions
```

Instead of:

```text
User1 → Policy
User2 → Policy
User3 → Policy
...
User100 → Policy
```

Do this:

```text
User1
User2
User3
...
User100
      │
      ▼
S3-Admins Group
      │
      ▼
One IAM Policy
```

Now if tomorrow you change the policy...

Only **one policy** changes.

All 100 users automatically receive the new permissions.

---

# Why IAM Group?

Groups are designed for:

* Managing many users
* Same permissions
* Easy administration

Example:

```
Developers Group

Read S3
Write S3
```

Every developer automatically gets those permissions.

---

# Why not IAM Role?

This confuses many students.

Remember:

### IAM Group

Used for

> **Users with same permissions**

Example

```
Developers
Finance
HR
Admins
```

---

### IAM Role

Used for

Temporary access

Examples

* EC2 accessing S3
* Lambda accessing DynamoDB
* Cross-account access
* Federation (SSO)
* Temporary credentials

Roles are **assumed**, not joined.

You don't add users permanently to a role for permission management.

---

# Why not EventBridge?

Could automate attaching policies.

But then:

```
100 Users

↓

100 Individual Policies
```

Still difficult to manage later.

AWS always prefers Groups.

---

# Why not Bucket Policy?

Bucket policies are resource-based policies.

The problem is **managing users**, not the bucket.

Also giving unlimited bucket access is against least privilege.

---

# Memory Trick

Whenever you see

* Many users
* Same permissions
* Easier management

Answer:

> **IAM Group**

---

Whenever you see

* EC2
* Lambda
* Cross Account
* Temporary credentials
* STS
* Federation

Answer:

> **IAM Role**

---

# Quick Comparison

| Service        | Used For                                                |
| -------------- | ------------------------------------------------------- |
| **IAM User**   | One individual person                                   |
| **IAM Group**  | Many users with same permissions                        |
| **IAM Role**   | Temporary permissions (EC2, Lambda, SSO, Cross-account) |
| **IAM Policy** | JSON document that defines permissions                  |

---

## SAA Exam Shortcut

Think of it this way:

* 👤 **One person** → IAM User
* 👥 **Many people, same permissions** → IAM Group ✅
* 🎭 **Temporary identity** → IAM Role
* 📜 **Permission document** → IAM Policy

This pattern appears repeatedly in the SAA exam, so recognizing **"many users + same permissions = IAM Group"** will help you answer these questions very quickly.
