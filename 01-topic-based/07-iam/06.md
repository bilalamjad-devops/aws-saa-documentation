6. Question

Category: CSAA – Design Secure Architectures

A company must integrate the Lightweight Directory Access Protocol (LDAP) directory service from the on-premises data center to the AWS VPC using IAM. The identity store which is currently being used is not compatible with SAML.

Which of the following provides the most valid approach to implement the integration?

- Use AWS IAM Identity Center to manage access between AWS and your LDAP.
- Use IAM roles to rotate the IAM credentials whenever LDAP credentials are updated.
- `Develop an on-premises custom identity broker application and use STS to issue short-lived AWS credentials.`
- Use an IAM policy that references the LDAP identifiers and AWS credentials.

This is one of the more conceptual IAM questions. The trick is to know **what happens when LDAP does not support SAML**.

---

# Step 1: Identify the keywords

The question says:

* On-premises **LDAP**
* Integrate with AWS
* **Not compatible with SAML**

The keyword is:

> **Not compatible with SAML**

This changes the whole answer.

---

# Normally (when SAML is supported)

```text
User
   │
   ▼
Active Directory / LDAP
   │
   ▼
SAML Identity Provider
   │
   ▼
AWS STS
   │
Temporary Credentials
   │
   ▼
AWS Resources
```

This is the standard federation architecture.

---

# But here...

The question explicitly says:

> LDAP is **NOT SAML compatible**

So this architecture cannot be used.

AWS documentation recommends:

> **Custom Identity Broker**

---

# Correct architecture

```text
Employee
      │
      ▼
LDAP
      │
      ▼
Custom Identity Broker
      │
      ▼
AWS STS
      │
Temporary Credentials
      │
      ▼
AWS Resources
```

The broker acts as a translator between your LDAP system and AWS.

---

# What does the Identity Broker do?

It performs three jobs:

### 1. Authenticate user

Checks the user against LDAP.

↓

### 2. Call STS

Uses:

* AssumeRole
  or
* GetFederationToken

↓

### 3. Return temporary credentials

The user receives:

* Access Key
* Secret Key
* Session Token

---

# Why STS?

Because AWS resources require AWS credentials.

LDAP credentials cannot directly access AWS.

STS converts:

```text
LDAP Identity

↓

Temporary AWS Credentials
```

---

# Why not IAM Identity Center?

Normally IAM Identity Center is a great answer.

But notice the keyword:

> **Identity store is not compatible with SAML**

IAM Identity Center primarily relies on standards such as SAML/OIDC for federation with external identity providers.

Since SAML compatibility is explicitly missing, AWS recommends a **custom identity broker** instead.

---

# Why not IAM Roles?

Roles only define permissions.

They don't authenticate LDAP users.

You still need something that talks to LDAP.

---

# Why not IAM Policy?

Policies only answer:

> What can you do?

They do not answer:

> Who are you?

Nor do they integrate LDAP.

---

# Memory Trick

### If LDAP supports SAML

Think:

```text
LDAP

↓

SAML

↓

STS

↓

AWS
```

---

### If LDAP does NOT support SAML

Think:

```text
LDAP

↓

Custom Identity Broker

↓

STS

↓

AWS
```

# Quick Table

| Situation                 | Solution                       |
| ------------------------- | ------------------------------ |
| AD/LDAP supports SAML     | Federation + STS               |
| LDAP doesn't support SAML | Custom Identity Broker + STS ✅ |
| Temporary AWS credentials | STS                            |
| Permissions               | IAM Role                       |

---

# SAA Exam Shortcut

Whenever you see:

* LDAP
* Active Directory
* **Not SAML compatible**

Immediately think:

> **Custom Identity Broker + STS**

---

## Easy way to remember

```
SAML available?
      │
   Yes ▼
Federation
      │
      ▼
STS

No
      │
      ▼
Identity Broker
      │
      ▼
STS
```

The key takeaway is that **STS is still used to issue temporary AWS credentials**. The only difference is **how the user is authenticated before reaching STS**:

* **SAML available** → SAML federation.
* **SAML unavailable** → Custom Identity Broker.
