4. Question

Category: CSAA – Design Secure Architectures

An Intelligence Agency developed a missile tracking application that is hosted on both development and production AWS accounts. The Intelligence agency’s junior developer only has access to the development account. The developer has received security clearance to access the agency’s production account, but the access is only temporary, and only write access to Amazon EC2 and Amazon S3 is allowed.

Which of the following allows the developer to issue short-lived access tokens that act as temporary security credentials, leveraging AWS Identity and Access Management (IAM) roles, to allow access to AWS resources?

- `Use AWS Security Token Service (STS)`
- Use Amazon Cognito to issue JSON Web Tokens (JWT)
- Use AWS IAM Identity Center
- All of the given options are correct.

This is another **STS** question. The exam loves asking about **temporary access**.

---

# Step 1: Identify the keywords

The scenario says:

* Development account
* Production account
* **Temporary access**
* Cross-account access
* **Short-lived security credentials**

The keyword is:

> **Temporary credentials**

Whenever you see this, immediately think:

> **AWS STS**

---

# Architecture

```text
Development Account
-------------------

Developer (IAM User)
        │
        │ AssumeRole
        ▼
AWS STS
        │
Temporary Credentials
        │
        ▼

Production Account
------------------

IAM Role
        │
Allows:
- EC2 Write
- S3 Write
```

---

# How it works

### Step 1

Developer logs into Development account.

↓

### Step 2

Developer calls

```text
AssumeRole
```

↓

### Step 3

AWS STS returns

* Temporary Access Key
* Temporary Secret Key
* Session Token

↓

### Step 4

Developer uses those credentials

↓

Access Production EC2 and S3.

---

# Why STS?

STS exists for exactly these scenarios:

* Cross-account access
* Temporary access
* Federation
* SSO
* AssumeRole

---

# Why not Cognito?

Many students confuse this.

Cognito is for

* Mobile apps
* Web applications
* User login

Example

```text
Login

↓

Username

↓

Password

↓

JWT Token
```

JWT authenticates the user.

It **does not** provide temporary AWS credentials for cross-account IAM access (unless used in broader identity federation flows, which is not what this question is asking).

Think:

> Cognito = Login users

---

# Why not IAM Identity Center?

IAM Identity Center provides

* Workforce login
* Centralized SSO
* Multiple AWS accounts
* Applications

It **doesn't generate temporary credentials directly**.

Behind the scenes it uses STS.

The question asks

> Which service issues temporary credentials?

Answer:

> STS

---

# Memory Trick

Whenever you see

* Temporary credentials
* Temporary tokens
* AssumeRole
* Cross Account
* Federation
* Short-lived access

Answer:

> **AWS STS**

---

# Quick Comparison

| Service                 | Purpose                                         |
| ----------------------- | ----------------------------------------------- |
| **STS**                 | Temporary AWS credentials                       |
| **IAM Role**            | Defines permissions that STS credentials assume |
| **Cognito**             | User authentication for web/mobile apps         |
| **IAM Identity Center** | Workforce SSO across AWS accounts/apps          |

---

# SAA Exam Shortcut

Look for these phrases:

* ✅ Temporary access
* ✅ Short-lived credentials
* ✅ Cross-account access
* ✅ AssumeRole
* ✅ Session token

→ **AWS STS**

If you instead see:

* Mobile app login
* User signup/signin
* Social login (Google, Facebook, Apple)

→ **Amazon Cognito**

If you see:

* Company employees
* Centralized SSO
* Multiple AWS accounts
* Workforce identities

→ **IAM Identity Center**

---

### One-line exam rule

> **STS creates temporary credentials. IAM Roles define what those temporary credentials are allowed to do.**
