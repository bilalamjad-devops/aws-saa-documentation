5. Question

Category: CSAA – Design Secure Architectures

A Solutions Architect created a brand new IAM User with a default setting using AWS CLI. This is intended to be used to send API requests to Amazon S3, DynamoDB, Lambda, and other AWS resources of the company’s cloud infrastructure.

Which of the following must be done to allow the user to make API calls to the AWS resources?

- Enable Multi-Factor Authentication for the user.
- `Create a set of Access Keys for the user and attach the necessary permissions.`
- Assign an IAM Policy to the user to allow it to send API calls.
- Do nothing as the IAM User is already capable of sending API calls to your AWS resources.

This question tests a small but very important AWS IAM detail.

---

# Step 1: Identify the keywords

The question says:

* IAM User
* **Created using AWS CLI**
* Needs to send **API requests**
* S3, Lambda, DynamoDB

The keyword is:

> **API requests**

To make API calls, an IAM user needs **credentials**.

---

# What does a new IAM User created via AWS CLI have?

By default:

* ❌ No console password
* ❌ No access keys
* ❌ No credentials

So even if you attach an IAM policy, the user **cannot authenticate** to AWS.

---

# Correct sequence

```text
Create IAM User
        │
        ▼
Create Access Keys
        │
        ▼
Attach IAM Policy
        │
        ▼
Use AWS CLI / SDK / API
```

Both are needed:

1. **Access Keys** → Authentication (Who are you?)
2. **IAM Policy** → Authorization (What are you allowed to do?)

---

# Why is the correct answer only "Create Access Keys"?

Because the question asks:

> **What must be done to allow the user to make API calls?**

The missing requirement is **credentials**.

Without Access Keys:

```text
aws s3 ls
```

AWS cannot identify the user.

No authentication.

---

# Authentication vs Authorization

This distinction is extremely important.

### Authentication

Who are you?

AWS uses:

* Access Key ID
* Secret Access Key

---

### Authorization

What can you do?

AWS checks:

* IAM Policies
* Resource Policies
* SCPs
* Permission Boundaries

---

Think of it like this:

```text
Authentication
↓

"I am Bilal."

↓

Authorization

"Bilal can read S3."
```

---

# Why not IAM Policy?

Many students choose this.

Suppose:

```text
IAM Policy

Allow S3
Allow DynamoDB
Allow Lambda
```

Great.

But...

How does AWS know **who** is making the request?

There are no Access Keys.

Authentication fails first.

---

# Why not MFA?

MFA protects logins.

It does **not** create API credentials.

Programmatic access uses Access Keys (and optionally session tokens via STS), not MFA alone.

---

# Why not "Do nothing"?

Wrong.

IAM Users created via CLI start with **no credentials**.

---

# Memory Trick

For an IAM User:

### Console Login

Need:

* Username
* Password

---

### CLI / SDK / API

Need:

* Access Key ID
* Secret Access Key

---

# Quick Table

| Access Method | Required Credential     |
| ------------- | ----------------------- |
| AWS Console   | Username + Password     |
| AWS CLI       | Access Key + Secret Key |
| SDK           | Access Key + Secret Key |
| REST API      | Access Key + Secret Key |

---

# SAA Exam Shortcut

Whenever you see:

* CLI
* SDK
* Programmatic access
* API calls

Immediately think:

> **Access Keys**

Whenever you see:

* Login to Console

Think:

> **Console Password**

---

### One-line exam rule

* **Access Keys = Authenticate programmatic requests (CLI/SDK/API).**
* **IAM Policies = Define what those authenticated requests are allowed to do.**

Both are needed in practice, but if the question asks what is **missing to make API calls**, the answer is **create Access Keys**.
