4. Question

Category: CSAA – Design High-Performing Architectures

A multinational company has been building its new data analytics platform with high-performance computing workloads (HPC) which requires a scalable, POSIX-compliant storage service. The data need to be stored redundantly across multiple AZs and allows concurrent connections from thousands of EC2 instances hosted on multiple Availability Zones.

Which of the following AWS storage service is the most suitable one to use in this scenario?

- Amazon EBS Volumes
- `Amazon Elastic File System`
- Amazon ElastiCache
- Amazon S3

This question is basically testing whether you recognize the **classic EFS keywords**. By now, you should notice that AWS keeps asking the same concept in different ways.

Let's decode it.

---

# Step 1: Highlight the keywords

The question says:

* ✅ **High-Performance Computing (HPC)**
* ✅ **POSIX-compliant**
* ✅ **Multiple AZs**
* ✅ **Concurrent connections**
* ✅ **Thousands of EC2 instances**
* ✅ **Scalable storage**

Honestly, after reading just **POSIX** or **thousands of EC2**, you should already be thinking:

> **Amazon EFS**

---

# Architecture

Without EFS:

```text
EC2-1 → EBS

EC2-2 → EBS

EC2-3 → EBS
```

Each server has its own disk.

---

With EFS:

```text
                Amazon EFS
          _____________________
         /      /      \       \
      EC2-1  EC2-2   EC2-3 ... EC2-1000
```

Everyone reads and writes to the **same filesystem**.

---

# Why HPC uses EFS

Imagine you're rendering a movie.

There are

```text
1000 EC2 instances
```

Every instance reads

```text
/shared/models/
```

and writes

```text
/shared/output/
```

If every EC2 had its own EBS,

they couldn't share data.

That's why HPC commonly uses **EFS** for shared storage.

---

# POSIX Again

This keyword appears again.

POSIX means your application expects normal Linux file operations.

Example:

```bash
mkdir

ls

touch

chmod

read

write
```

Only a filesystem provides this.

EFS does.

---

# Why not EBS?

Remember:

EBS = Block Storage

Normally

```text
One EC2

↓

One EBS
```

It isn't designed for:

* thousands of EC2
* shared filesystem
* multiple AZs

---

# Why not S3?

Students often choose S3 because:

> "It stores files."

Yes.

But the application needs

```text
/shared/data/file.txt
```

with

* directories
* file locking
* POSIX semantics
* NFS

S3 is object storage, accessed through APIs—not a shared Linux filesystem.

---

# Why not ElastiCache?

ElastiCache stores data in RAM.

Example:

```text
User session

↓

Cached
```

Not

```text
/shared/report.pdf
```

Wrong service.

---

Notice how all four EFS questions ask the same thing in different words.

| Question Keyword      | Answer |
| --------------------- | ------ |
| POSIX                 | EFS    |
| NFS                   | EFS    |
| Shared storage        | EFS    |
| Multiple EC2          | EFS    |
| Multiple AZ           | EFS    |
| Thousands of clients  | EFS    |
| Concurrent access     | EFS    |
| HPC shared filesystem | EFS    |

# Quick Comparison

| Service     | Storage Type | Shared | Multiple AZ | Linux File System |
| ----------- | ------------ | ------ | ----------- | ----------------- |
| EBS         | Block        | ❌      | ❌           | ❌                 |
| S3          | Object       | ❌      | ✅           | ❌                 |
| ElastiCache | Memory       | ❌      | N/A         | ❌                 |
| **EFS**     | **File**     | ✅      | ✅           | ✅                 |

---

# SAA Memory Rule ⭐⭐⭐⭐⭐

Whenever you see a combination like:

* **HPC**
* **POSIX**
* **NFS**
* **Thousands of EC2**
* **Concurrent access**
* **Shared filesystem**
* **Multiple Availability Zones**

don't spend time comparing services.

The AWS architect's answer is almost always:

> ✅ **Amazon EFS**.

This is one of the highest-yield patterns for the SAA exam.
