3. Question

Category: CSAA – Design High-Performing Architectures

A leading e-commerce company is in need of a storage solution that can be simultaneously accessed by 1000 Linux servers in multiple availability zones. The servers are hosted in EC2 instances that use a hierarchical directory structure via the NFSv4 protocol. The service should be able to handle the rapidly changing data at scale while still maintaining high performance. It should also be highly durable and highly available whenever the servers will pull data from it, with little need for management.

As the Solutions Architect, which of the following services is the most cost-effective choice that you should use to meet the above requirement?

- Amazon S3
- Amazon FSx for Windows File Server
- Amazon EBS
- `Amazon EFS`

Perfect. This question is mainly testing whether you can distinguish **EFS vs S3 vs EBS vs FSx**. Once you identify the keywords, the answer becomes obvious.

---

# Step 1: Highlight the keywords

The question says:

* ✅ **1000 Linux servers**
* ✅ **Multiple Availability Zones**
* ✅ **NFSv4 protocol**
* ✅ **Hierarchical directory structure**
* ✅ **Rapidly changing data**
* ✅ **Highly available**
* ✅ **Little management**

These keywords all point to **Amazon EFS**.

---

# Keyword 1: 1000 Linux Servers

Can EBS be shared with 1000 EC2 instances?

❌ No.

Can EFS?

✅ Yes.

Think:

```text
        Amazon EFS
      /   |   |   |   \
EC2  EC2 EC2 EC2 ... EC2
      (1000 Linux servers)
```

EFS is designed exactly for this.

---

# Keyword 2: NFSv4

This is one of the easiest exam clues.

NFS = **Network File System**

AWS file service that supports NFS:

✅ Amazon EFS

Remember:

* **EFS → NFS (Linux)**
* **FSx Windows → SMB (Windows)**

---

# Keyword 3: Hierarchical Directory

The question mentions:

> hierarchical directory structure

That means folders like:

```text
/shared
    /images
    /videos
    /documents
```

That is a **filesystem**.

S3 doesn't have real folders—it stores objects with key prefixes.

So:

❌ S3

✅ EFS

---

# Keyword 4: Rapidly Changing Data

Example:

```text
file.txt

↓

updated

↓

updated again

↓

updated again
```

EFS handles frequent reads/writes very well.

---

# Why not S3?

S3 is amazing for:

* Images
* Videos
* Backups
* Static websites
* Archives

But not for applications that need:

* file locking
* POSIX
* NFS
* shared filesystem semantics

---

# Keyword 5: Multiple Availability Zones

EFS automatically stores data across multiple AZs (Regional EFS).

EBS does not.

Remember:

**EBS**

```text
AZ-A

EC2

↓

EBS
```

**EFS**

```text
AZ-A        AZ-B       AZ-C

EC2         EC2        EC2
   \         |         /
      Amazon EFS
```

---

# Why not FSx for Windows?

Question says:

> Linux servers

FSx for Windows is for:

* SMB
* Active Directory
* Windows applications

Wrong operating system.

---

# Why not EBS?

EBS is:

* Block storage
* One EC2 (normally)
* Same AZ

Not:

* Shared
* NFS
* Thousands of Linux servers

# Memory Trick

Imagine you're working in an office:

* **EBS** = Your personal laptop's hard drive. Only you use it.
* **EFS** = A shared company network drive (`\\shared` or `/mnt/shared`) that everyone can access.
* **S3** = A cloud storage bucket where you upload/download files via an API, not as a mounted Linux filesystem.
* **FSx for Windows** = A Windows file server using SMB for Windows clients.

When you see combinations like:

* **Linux**
* **NFS**
* **POSIX**
* **Shared storage**
* **Multiple EC2 instances**
* **Multiple AZs**
* **Thousands of clients**

don't overthink it.

Your immediate answer should be:

**Amazon EFS**.
