
2. Question

Category: CSAA – Design High-Performing Architectures

A content management system (CMS) is hosted on a fleet of auto-scaled, On-Demand Amazon EC2 instances that use Amazon Aurora as its database. Currently, the system stores the file documents that users upload in one of the attached Amazon EBS volumes. The system’s performance has been observed to be slow, and the manager has instructed the team to improve the architecture.

In this scenario, which solution should be implemented to achieve a scalable, highly available, POSIX-compliant shared file system?

- Upgrade your existing EBS volumes to Provisioned IOPS SSD volumes
- Create an Amazon S3 bucket and use this as the storage for the CMS
- Leverage Amazon ElastiCache to cache frequently accessed data and reduce latency
- `Use Amazon EFS to provide a shared file system for concurrent access to data`

# Step 1: Read the architecture

Current architecture:

```text
                Users
                  │
          Application Load Balancer
                  │
          Auto Scaling Group
          ┌────────┬────────┬────────┐
          │ EC2-1  │ EC2-2  │ EC2-3  │
          └────────┴────────┴────────┘
             │         │         │
            EBS       EBS       EBS
                  │
             Aurora Database
```

Users upload files.

Where are they stored?

👉 On an **EBS volume**.

---

# Step 2: What's the problem?

Suppose User A uploads:

```text
resume.pdf
```

It is saved on

```text
EC2-1
```

Now another user accesses the website.

The ALB sends him to

```text
EC2-2
```

Can EC2-2 see the file?

❌ No.

Because

```text
EC2-1
   │
 EBS-1
```

and

```text
EC2-2
   │
 EBS-2
```

are different disks.

Every server has its own storage.

---

# What do we need?

The question says

> scalable

> highly available

> shared

> POSIX-compliant

These four words point directly to one service.

---

# POSIX-Compliant ⭐⭐⭐⭐⭐

This is one of the biggest exam keywords.

POSIX means

A **standard Linux file system**.

Your application can do things like:

```bash
open()

read()

write()

chmod()

mkdir()

rm
```

just like any Linux filesystem.

EFS supports this.

S3 does NOT.

---

# Shared File System

Instead of

```text
EC2-1 → EBS-1

EC2-2 → EBS-2

EC2-3 → EBS-3
```

Use

```text
          EFS
      /     |     \
     /      |      \
 EC2-1   EC2-2   EC2-3
```

Now

User uploads

```text
resume.pdf
```

to

```text
EC2-1
```

EC2-2 immediately sees it.

EC2-3 immediately sees it.

That is exactly what a CMS needs.

---

# Why not EBS?

Can EBS be faster?

Yes.

Provisioned IOPS is faster.

But...

The problem isn't speed.

The problem is

Shared storage.

Even a very fast EBS

is still attached to

one instance (or Multi-Attach only in limited same-AZ scenarios).

It doesn't solve the CMS shared-files problem.

---

# Why not S3?

This is the biggest trap.

Students think

Files

↓

S3

But

CMS applications usually expect

```bash
/var/www/uploads/
```

which is a filesystem.

S3 is

Object Storage

NOT

File Storage.

It doesn't behave like Linux storage.

---

# Why not ElastiCache?

ElastiCache

stores

```text
Key → Value
```

in memory.

Example

```
Session data

Shopping cart

Database cache
```

Not

```
resume.pdf

photo.jpg

invoice.docx
```

Wrong service.

---

# Exam Trick

Whenever AWS says

> POSIX-compliant

Immediately think

✅ Amazon EFS

Because AWS almost never uses the word "POSIX" unless the answer is EFS.

---

# Quick Comparison

| Service     | Type             | Shared?            | POSIX?                                                                 |
| ----------- | ---------------- | ------------------ | ---------------------------------------------------------------------- |
| EBS         | Block Storage    | ❌ Normally No      | Filesystem only after formatting, but not shared across many instances |
| S3          | Object Storage   | ❌ Not a filesystem | ❌                                                                      |
| ElastiCache | In-memory cache  | ❌                  | ❌                                                                      |
| **EFS**     | **File Storage** | ✅ Yes              | ✅ Yes                                                                  |

---

# Architecture After Solution

```text
              Users
                 │
               ALB
                 │
       Auto Scaling Group
     ┌──────┬──────┬──────┐
     │EC2-1 │EC2-2 │EC2-3 │
     └──┬───┴──┬───┴──┬───┘
         │      │      │
         └──────┼──────┘
                │
               EFS
                │
             Aurora DB
```

Now every EC2 instance sees the **same uploaded files**, making the CMS scalable and highly available.

---

## ⭐ SAA Memory Rule

If you see:

* Auto Scaling Group
* Multiple EC2 instances
* CMS / WordPress / Drupal / Joomla
* Shared uploads
* Shared documents
* Shared configuration
* **POSIX-compliant**
* **Concurrent access**
* Shared Linux filesystem

👉 The answer is almost always **Amazon EFS**.
