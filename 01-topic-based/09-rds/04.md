4. Question

Category: CSAA – Design Resilient Architectures

An accounting application uses an Amazon RDS database configured with RDS Multi-AZ to improve availability. What would happen to RDS if the primary database instance fails?

- `The canonical name record (CNAME) is switched from the primary to standby instance.`
- The IP address of the primary DB instance is switched to the standby DB instance.
- The primary database instance will reboot.
- A new database instance is created in the standby Availability Zone.



Bilal, this is actually one of the **easiest Multi-AZ questions** once you know how failover works. Let's visualize it.

---

# Before Failure

Imagine two databases.

```text
Availability Zone A

Primary DB
```

```text
Availability Zone B

Standby DB
```

Your application connects like this:

```text
Application
      │
      ▼
database.company.com
      │
      ▼
Primary DB
```

Notice something important.

The application **does not connect directly** to the database's IP.

It connects to:

```text
database.company.com
```

This is called the **RDS Endpoint**.

---

# Then Primary Crashes 💥

```text
Primary DB ❌
```

Now AWS says:

"No problem."

The standby is already there.

```text
Standby
↓

Becomes Primary
```

---

# But How Does the Application Know?

Does AWS change the application?

No.

Does AWS change the database IP?

No.

AWS changes where the **endpoint name** points.

Think of it like this:

Before:

```text
database.company.com
        │
        ▼
Primary
```

After failover:

```text
database.company.com
        │
        ▼
Standby
```

The **endpoint stays the same**.

Only its DNS record changes.

AWS changes the **CNAME (DNS)** behind the scenes.

---

# Real-Life Analogy

Suppose your friend is saved in your phone as:

```text
Ali
```

Today:

```text
Ali → 0300-1111111
```

Tomorrow Ali changes his phone number.

Instead of you changing everything,

your contacts app updates:

```text
Ali → 0300-9999999
```

You still call **Ali**.

You don't care about the number.

The **RDS endpoint** works the same way.

---

# Why Not IP Address?

AWS databases are in different Availability Zones.

Each AZ has different networking.

The IP changes.

Instead of remembering IPs, applications always use the **DNS endpoint**.

AWS simply updates the DNS.

---

# Why the Other Answers Are Wrong

### ❌ IP address is switched

AWS does **not** move an IP between AZs.

Instead, it updates the DNS (CNAME).

---

### ❌ Primary reboots

The question says:

> Primary **fails**.

If it has failed, rebooting it is not the failover process.

AWS immediately promotes the standby.

---

### ❌ New database is created

No.

In **Multi-AZ**, the standby already exists.

```text
Primary (AZ-1)

Standby (AZ-2)
```

AWS does **not** create a new one during failover.

It simply promotes the existing standby.

---

# One-Line Memory Trick

Think:

**Multi-AZ = Two databases already exist.**

If one dies:

> AWS changes the **DNS (CNAME)** to point to the standby.

---

## Exam Shortcut

When you see:

* Multi-AZ
* Failover
* Primary fails
* Standby takes over

Immediately think:

**AWS updates the RDS endpoint (CNAME/DNS), not the IP address.**

---

### A small note

You don't need to become a DNS expert for the exam.

Just remember this simple rule:

* **Application connects to the RDS endpoint (DNS name).**
* **During failover, AWS changes the DNS/CNAME to point to the standby database.**
* **Your application keeps using the same endpoint.**

That's exactly why applications can reconnect with minimal changes after a Multi-AZ failover.
