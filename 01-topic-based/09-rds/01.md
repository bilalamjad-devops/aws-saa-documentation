1. Question

Category: CSAA – Design Resilient Architectures

An application that records weather data every minute is deployed in a fleet of Amazon EC2 Spot instances and uses a MySQL RDS database instance. Currently, there is only one Amazon RDS instance running in one Availability Zone. The database needs to be improved to ensure high availability by enabling synchronous data replication to another RDS instance.

Which of the following performs synchronous data replication in RDS?

- Amazon CloudFront running as a Multi-AZ deployment
- RDS Read Replica
- `RDS DB instance running as a Multi-AZ deployment`
- Amazon DynamoDB Read Replica

Bilal, this is exactly where many people struggle with RDS. The problem isn't RDS—it's that AWS uses similar terms that sound alike. Let's forget AWS for a minute and use a real-life example.

# Imagine You Own a Bank 🏦

You have one accountant.

Every customer comes to him.

Now imagine two different situations.

---

# Situation 1: Accountant Gets Sick

Your bank cannot stop.

So you hire **another accountant**.

```
Primary Accountant  ---> Working

Backup Accountant   ---> Waiting
```

Whenever the primary accountant writes something:

```
Primary writes

↓

Backup writes immediately

↓

Customer leaves
```

Both accountants always have the same records.

If the primary accountant gets sick:

```
Primary ❌

↓

Backup immediately starts working
```

Customers don't even notice.

### This is **RDS Multi-AZ**

Purpose:

> Keep the application running.

Keyword:

✅ High Availability

---

# Situation 2: Too Many Customers

Your accountant is healthy.

But there are 10,000 customers asking:

> "What is my balance?"

Reading takes time.

Instead of hiring a backup accountant...

You hire another accountant only to answer questions.

```
Primary Accountant

        │
        │
        ▼

Read Accountant
```

Now

Customer 1

"What is my balance?"

↓

Read Accountant

Customer 2

"Show my statement"

↓

Read Accountant

Customer 3

"Last month's transactions"

↓

Read Accountant

The primary accountant only handles deposits and withdrawals.

### This is **Read Replica**

Purpose:

> Make reading faster.

---

## Multi-AZ

---

## Read Replica

Question:

Answer:

Use **Read Replica**.

If the question says...

**High Availability** → ✅ Multi-AZ

**Automatic Failover** → ✅ Multi-AZ

# Important Concepts

## 1. RDS Multi-AZ = High Availability

Purpose:

* Protect against AZ failure
* Automatic failover
* Disaster recovery

Replication:

* **Synchronous**

Architecture:

```
Primary DB (AZ-A)
        │
        │  Synchronous Replication
        ▼
Standby DB (AZ-B)
```

If Primary fails:

```
Primary ❌

↓

Standby becomes Primary automatically
```

Application reconnects automatically.

---

## 2. Read Replica = Read Scaling

Purpose:

* Increase read performance
* Reporting
* Analytics

Replication:

* **Asynchronous**

Architecture

```
Primary
      │
      │ Async Replication
      ▼
Read Replica
```

Can have multiple replicas.

Used for:

* Heavy SELECT queries
* Reporting
* BI dashboards

NOT for HA.

# Synchronous vs Asynchronous

### Multi-AZ

```
Application

↓

Primary writes

↓

Standby also writes

↓

Only then success returned
```

Pros

* No data loss

Cons

* Slightly slower writes

---

### Read Replica

```
Primary writes

↓

Success returned immediately

↓

Replica updated later
```

Pros

* Fast

Cons

* Small replication lag

# Wrong Options

### ❌ RDS Read Replica

Wrong because:

* Replication is **asynchronous**
* Used for **read scaling**, not HA.

---

### ❌ CloudFront

CloudFront is a CDN.

It caches web content at edge locations.

Nothing to do with databases.

---

### ❌ DynamoDB Read Replica

DynamoDB doesn't use "Read Replicas."

For cross-Region replication it uses:

* **Global Tables**
