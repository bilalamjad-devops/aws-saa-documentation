6. Question

Category: CSAA – Design Cost-Optimized Architectures

A company is building an internal application that allows users to upload images. Each upload request must be sent to Amazon Kinesis Data Streams for processing before the pictures are stored in an Amazon S3 bucket.

The application should immediately return a success message to the user after the upload, while the downstream processing is handled asynchronously. The processing typically takes about 5 minutes to complete.

Which solution will enable asynchronous processing from Kinesis to S3 in the most cost-effective way?

- `Use Kinesis Data Streams with AWS Lambda consumers to asynchronously process records and write them to S3.`
- Use a combination of AWS Lambda and Step Functions to orchestrate service components and asynchronously process the requests.
- Use a combination of Amazon SQS to queue the requests and then asynchronously process them using On-Demand Amazon EC2 Instances.
- Send data from Kinesis Data Streams to Amazon Data Firehose and configure it to deliver directly to S3.

### Keywords

---

# Correct Architecture

```text
User
   │
Upload
   │
   ▼
Kinesis Data Streams
   │
   ▼
Lambda Consumer
   │
(Process ~5 minutes)
   │
   ▼
Amazon S3
```

The application responds immediately after writing to **Kinesis**.

The actual processing happens later by **Lambda**, so it is **asynchronous**.

---

# Why Lambda Consumer?

Lambda has native integration with **Kinesis Data Streams** using an **Event Source Mapping**.

Lambda automatically:

* Polls Kinesis
* Reads batches of records
* Invokes your function
* Handles retries
* Manages checkpoints

You don't need to manage any servers.

---

## Why This Is Cost-Effective

* Serverless
* Auto scales
* No EC2
* No infrastructure management
* Pay only for:

  * Kinesis shards
  * Lambda execution

---

# Why Others Are Wrong

## ❌ Step Functions

Step Functions are used for:

* Multi-step workflows
* Branching
* Parallel execution
* Human approvals
* Complex orchestration

Example:

```text
Upload
   ↓
Validate
   ↓
Resize
   ↓
Notify
   ↓
Archive
```

This question only has **one processing step**, so Step Functions add unnecessary cost.

---

## ❌ SQS + EC2

Problems:

* Requirement explicitly says **must use Kinesis**
* EC2 requires:

  * Provisioning
  * Scaling
  * Patching
  * Monitoring

Higher operational overhead.

---

## ❌ Kinesis → Data Firehose → S3

Firehose is designed for **delivery**, not complex processing.

Good for:

```text
Logs
Metrics
Clickstream
Streaming data

↓

S3
Redshift
OpenSearch
Splunk
```

Not ideal for:

* Long custom processing
* Business logic
* 5-minute processing

Lambda consumers provide much more flexibility.

---

# Exam Tips

### If you see:

* **Kinesis Data Streams**
* **Real-time processing**
* **Custom logic**
* **Asynchronous**

➡️ Think **Lambda Consumer**

---

### If you see:

* Deliver streaming data directly to S3
* Minimal transformation
* Managed delivery pipeline

➡️ Think **Kinesis Data Firehose**

---

### If you see:

* Multi-step workflow
* Retry logic
* Branching
* Sequential tasks
* Human approval

➡️ Think **Step Functions**

# Quick Comparison

| Requirement                    | Service                |
| ------------------------------ | ---------------------- |
| Real-time stream processing    | ✅ Kinesis Data Streams |
| Consume Kinesis records        | ✅ Lambda Consumer      |
| Deliver stream directly to S3  | ✅ Data Firehose        |
| Complex workflow/orchestration | ✅ Step Functions       |
| Queue messages                 | ✅ SQS                  |
| Server management              | ❌ EC2                  |
