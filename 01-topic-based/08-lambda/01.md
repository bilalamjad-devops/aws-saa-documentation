
1. Question

Category: CSAA – Design Resilient Architectures

An application is using an AWS Lambda function to process complex financial data that runs for 15 minutes on average. The Lambda function has been configured with the maximum memory allocation to handle the computational demands of the financial data processing. Most invocations were successfully processed. However, a few terminated invocations were noticed throughout the day, which caused a data discrepancy in the application.

Which of the following is the most likely cause of this issue?

- The concurrent execution limit has been reached.
- The failed Lambda Invocations contain a ServiceException error which means that the Lambda service encountered an internal error.
- `The failed Lambda functions have been running for over 15 minutes and reached the maximum execution time.`
- The Lambda function contains a recursive code and has been running for over 15 minutes.







This question is testing **one of the most common Lambda limits**.

---

# Keywords

* Lambda
* Runs **15 minutes**
* Some executions terminate
* Data discrepancy

Immediately think:

👉 **Lambda Timeout**

---

# Important Limit

AWS Lambda maximum execution time

* Default = **3 seconds**
* Maximum = **900 seconds**
* 900 seconds = **15 minutes**

**Exam fact**

> Lambda can never run longer than **15 minutes**.

---

# Scenario

Most executions

↓

Finish within 15 minutes

Some executions

↓

Take slightly longer

↓

Lambda automatically terminates them

↓

Incomplete processing

↓

Data inconsistency

---

# Correct Answer

✅ **The failed Lambda functions have been running for over 15 minutes and reached the maximum execution time.**

---

# Why others are wrong

## Concurrent execution limit

If concurrency limit reached

↓

Lambda is

**Throttled**

NOT

Terminated

Keyword:

Throttle ≠ Timeout

---

## ServiceException

Possible

But rare.

Would indicate AWS internal issue.

Not the best answer.

---

## Recursive Lambda

Recursive functions

↓

Create repeated invocations

↓

High costs

↓

Eventually throttled

Not simply terminated because of recursion.

---

# High-Yield Lambda Limits

| Feature           | Limit                    |
| ----------------- | ------------------------ |
| Default timeout   | 3 sec                    |
| Maximum timeout   | **15 min (900 sec)**     |
| Memory            | 128 MB – 10,240 MB       |
| Pricing           | Pay only while code runs |
| Server management | None                     |

# Exam Trick

Whenever you see:

* 15 minutes
* 900 seconds
* Long-running process
* Lambda suddenly stops

Answer:

✅ **Maximum Lambda execution timeout reached**

---

## Memory Trick

> **Lambda = Short-lived compute**
>
> **Maximum execution = 15 minutes (900 seconds)**

If a workload regularly exceeds 15 minutes, the exam often expects you to consider another compute service (such as EC2, ECS, Batch, or Step Functions coordinating longer-running work), because Lambda cannot execute beyond its hard timeout.
