4. Question

Category: CSAA – Design Cost-Optimized Architectures

A wellness company is currently working on a wearable device that monitors key health metrics such as heart rate, sleep, and steps per day. The device is designed to send data to an Amazon S3 bucket for storage and analysis. On a daily basis, the device produces 1 MB of data. In order to quickly process and summarize this data, the company requires 512 MB of memory and must complete the task within a maximum of 10 seconds.

Which solution can fulfill these requirements in the MOST cost-effective manner?

- Create an AWS Glue PySpark job to process the data.
- `Use AWS Lambda with a Python library for processing.`
- Use Amazon Data Firehose to send the data from the device to Amazon S3. Process the data on an EC2 instance with at least 512 MB of memory.
- Store the data in Amazon Redshift and process it with AWS Lambda.

Here are the **high-yield AWS Lambda notes** from these questions. This is enough for the SAA exam.

---

# AWS Lambda Summary (SAA-C03)

## 1. Maximum Execution Time (Timeout)

* **Default timeout:** 3 seconds
* **Maximum timeout:** **900 seconds (15 minutes)**

### Exam Trick

If a Lambda function:

* runs for ~15 minutes
* terminates unexpectedly
* some invocations fail

➡️ **Answer:** Lambda reached the **15-minute execution limit**.

**Not:**

* Concurrent execution limit
* ServiceException
* Recursive code

---

## 2. Lambda Pricing

You pay only for:

* Number of requests
* Execution duration (milliseconds)
* Memory allocated

No charge when Lambda is idle.

---

## 3. Lambda + S3

Very common architecture.

```
S3 Upload
     ↓
 Lambda Trigger
     ↓
Process Image/File/Data
```

Typical uses:

* Resize images
* Process CSV
* Generate thumbnails
* Convert files
* Parse logs

---

## 4. Lambda is Best for Small, Event-Driven Processing

Example:

* 1 MB/day data
* Needs 512 MB RAM
* Finish within 10 seconds

✅ Use **Lambda**

Why?

* Cheapest
* No servers
* Auto scaling
* Native S3 integration

Do NOT use:

* Glue
* EC2
* Redshift

Those are overkill.

# Lambda@Edge

Very important exam topic.

### Runs Lambda code at CloudFront Edge Locations

Instead of executing in one AWS Region,
Lambda executes **near the user**.

Benefits:

* Low latency
* Global processing
* No infrastructure management

---

## Lambda@Edge Use Cases

### 1. Modify HTTP Requests

Example:

```
User Request
      ↓
Lambda@Edge
      ↓
Modify request
      ↓
Origin
```

---

### 2. Modify HTTP Responses

Example:

Add

```
X-Company: ABC
```

to every response.

---

### 3. Device-Based Content

Example:

If browser supports:

```
WebP
```

Return:

```
image.webp
```

Else return

```
image.jpg
```

Lambda@Edge checks:

```
User-Agent
```

and serves the proper image.

---

### 4. Personalization

Can customize content based on:

* Country
* Device
* Browser
* Cookies
* Headers

# Lambda@Edge + Kinesis

Exam favorite.

Scenario:

Need

* clickstream analytics

* users worldwide
* low latency

Architecture:

```
Users

↓

CloudFront

↓

Lambda@Edge

↓

Kinesis

↓
```

Lambda@Edge

* processes near users

Kinesis

* real-time streaming

S3

* durable storage

# What NOT to Choose

Instead of Lambda@Edge:

❌ Route 53

Reason:

* Only DNS routing
* Cannot execute code

---

❌ CloudFront Response Headers Policy

Can:

* add/remove headers

Cannot:

* run logic
* inspect User-Agent
* select WebP/JPEG dynamically

---

❌ EC2

Higher operational overhead.

Use Lambda unless continuous compute is required.

---

❌ Athena

Athena is for **SQL queries on S3**, **not** real-time stream processing.
