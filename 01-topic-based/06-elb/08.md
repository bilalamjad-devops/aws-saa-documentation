
8. Question

Category: CSAA – Design High-Performing Architectures

A company plans to design a highly available architecture in AWS. They have two target groups with three EC2 instances each, which are added to an Application Load Balancer. 
In the security group of the EC2 instance, you have verified that port 80 for HTTP is allowed. However, the instances are still showing out of service from the load balancer.

What could be the root cause of this issue?

- The wrong subnet was used in your VPC
- The wrong instance type was used for the EC2 instance.
- `The health check configuration is not properly defined.`
- The instances are using the wrong AMI.

This is a classic **ALB troubleshooting** question.

---

# Keywords

* ALB
* Target Group
* EC2
* Security Group allows port 80
* **Instances are OutOfService**

The keyword is:

> **OutOfService**

Immediately think:

> **Health Check failed**

---

```text

     ▼
```



**Unhealthy (OutOfService)**

# Why is the Security Group not the problem?

The question already tells you:

> Port 80 is allowed.

So AWS is hinting:

> **Security Group is NOT the issue.**

This is a common exam technique—AWS eliminates one possible cause for you.

# Why not Wrong AMI?

A wrong AMI **could** prevent your app from running, but the exam asks for the **most likely ALB-related cause**.

The ALB marks instances as OutOfService because they **failed health checks**.

---

# Why not Wrong Instance Type?

Instance type affects:

* CPU
* Memory
* Performance

It does **not directly cause** an ALB to mark a target unhealthy.

---

# Why not Wrong Subnet?

If the EC2 instances were in an incorrect subnet, they might not even register correctly or be reachable, but the scenario specifically points to registered instances that are **OutOfService**. The most direct explanation is a failed health check.

---

# SAA Exam Tip

Whenever you see:

* Target Group
* OutOfService
* Unhealthy Targets
* ALB cannot reach EC2
* Security Group already correct

➡️ First suspect the **Health Check configuration**.

---

# ALB Health Check Checklist

When ALB marks a target unhealthy, think:

* ✅ Is the **health check path** correct (`/`, `/health`, etc.)?
* ✅ Is the **health check port** correct?
* ✅ Is the application listening on that port?
* ✅ Does it return an expected success status code (typically `200`)?
* ✅ Is the application responding within the health check timeout?
* ✅ Does the EC2 security group allow traffic from the ALB on the application port?

---

## Memory Shortcut

> **Target Group + OutOfService = Check Health Check first.**

This shortcut answers a large number of ALB troubleshooting questions in the SAA exam.
