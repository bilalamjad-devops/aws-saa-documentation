3. Question

Category: CSAA – Design Secure Architectures

A social media company needs to capture the detailed information of all HTTP requests that went through its public-facing Application Load Balancer every five minutes. The client’s IP address and network latencies must also be tracked, while the load balancer’s availability is monitored separately through the ELB health check. The captured data should be used for analyzing traffic patterns and for troubleshooting Docker applications orchestrated by the Amazon ECS Anywhere service.

Which of the following options meets the customer requirements with the LEAST amount of overhead?

- Enable AWS CloudTrail for their Application Load Balancer. Use the CloudTrail Lake to analyze and troubleshoot the application traffic.
- Install and run the AWS X-Ray daemon on the ECS cluster. Use the Amazon CloudWatch ServiceLens to analyze the traffic that goes through the application.
- `Enable access logs on the Application Load Balancer. Integrate the ECS cluster with Amazon CloudWatch Application Insights to analyze traffic patterns and simplify troubleshooting.`
- Integrate Amazon EventBridge metrics on the Application Load Balancer to capture the client IP address. Use Amazon CloudWatch GetMetricData to retrieve and analyze traffic-related metrics.



This is a **classic AWS observability question**. The trick is to know the difference between **CloudTrail, Access Logs, CloudWatch Metrics, and X-Ray**.

---

# Step 1: Read the Requirements

The company wants:

* ✅ Every HTTP request
* ✅ Client IP
* ✅ Network latency
* ✅ Request path
* ✅ Analyze traffic
* ✅ Troubleshoot

The biggest keyword is:

> **Detailed information of all HTTP requests**

Whenever you see that...

Think immediately:

> **ALB Access Logs**

---

# What are ALB Access Logs?

Every request that reaches your Application Load Balancer is recorded.

Example

```text
User
 │
 ▼
Application Load Balancer
 │
 ▼
EC2 / ECS
```

ALB stores logs like:

```text
Time

Client IP

HTTP Method

URL

Latency

Status Code

Target

Bytes Sent

Bytes Received
```

Example:

```text
10:00

IP: 45.12.3.9

GET /login

200 OK

Latency: 15 ms
```

Exactly what the question asks.

---

# Where are Access Logs Stored?

In

> **Amazon S3**

Not CloudWatch.

Remember:

```text
ALB

↓

Access Logs

↓

S3
```

---

# Why CloudWatch Application Insights?

The second requirement says:

> simplify troubleshooting

CloudWatch Application Insights automatically monitors applications.

It can detect

* application failures
* dependency issues
* infrastructure problems
* unusual behavior

So:

```text
ALB Access Logs
        +
CloudWatch Application Insights
```

is the best combination.

---

# Why Not CloudTrail?

Many students confuse this.

CloudTrail records

AWS API calls.

Example

```text
Create Load Balancer

Delete EC2

Modify Security Group

Create IAM User
```

CloudTrail does **NOT** record

```text
User visited

/login

/product

/search
```

CloudTrail knows

> "Someone modified the ALB."

It does NOT know

> "Bilal visited /login from IP 1.2.3.4"

---

# Why Not X-Ray?

X-Ray traces requests inside your application.

Example

```text
User

↓

ALB

↓

API

↓

Lambda

↓

DynamoDB
```

It shows

* where time was spent
* slow services
* service map

But

The question specifically wants

* Client IP
* Request details
* HTTP access logs

Those come from

**ALB Access Logs**

not X-Ray.

---

# Why Not EventBridge?

EventBridge works with

events.

Example

```text
EC2 stopped

IAM user created

S3 object uploaded

CodePipeline failed
```

It is **not** used for logging every HTTP request.

---

# ELB Health Check

The question also mentions:

> load balancer availability is monitored separately through ELB health check

That is just background information.

Health Checks determine if targets are healthy.

Example

```text
ALB

↓

GET /health

↓

200 OK

Healthy
```

If

```text
500

Timeout

Connection Refused
```

↓

Target becomes unhealthy.

This has nothing to do with Access Logs.

---

# Quick Comparison

| Service                | Purpose                                                                  |
| ---------------------- | ------------------------------------------------------------------------ |
| **ALB Access Logs**    | Logs every HTTP request (IP, URL, latency, status code)                  |
| **CloudTrail**         | Logs AWS API calls and account activity                                  |
| **CloudWatch Metrics** | CPU, Requests, Errors, Latency (aggregated metrics, not request details) |
| **X-Ray**              | Distributed tracing inside applications                                  |
| **EventBridge**        | Event routing and automation                                             |

## ⭐ SAA Exam Tip

A very common distinction tested is:

* **ALB Access Logs** = **individual request details** (client IP, URL, latency, status code, etc.).
* **CloudWatch Metrics** = **aggregated numbers** (e.g., RequestCount, 5XX errors, Average Latency).
* **CloudTrail** = **AWS management/API activity**, not application traffic.

When the question asks for **every HTTP request**, **client IP**, or **request-level analysis**, the answer is almost always **Enable ALB Access Logs**.
