3. Question

Category: CSAA – Design Secure Architectures

A company developed a meal planning application that provides meal recommendations for the week, as well as the food consumption of the users. The application resides on an Amazon EC2 instance, which requires access to various AWS services for its day-to-day operations.

Which of the following is the best way to allow the EC2 instance to access the Amazon S3 bucket and other AWS services?

- Add the API Credentials in the Security Group and assign it to the EC2 instance.
- Store the API credentials in a bastion host.
- `Create a role in IAM and assign it to the EC2 instance.`
- Use BatchGetSecretValue to retrieve long-term API credentials from AWS Secrets Manager and store them on the EC2 instance.

This is one of the **highest-frequency SAA questions**. If you remember only one IAM rule for EC2, make it this:

> **Never store AWS access keys on an EC2 instance. Attach an IAM Role instead.**

Let's break it down.

---

# Step 1: Read the question

> EC2 instance requires access to Amazon S3 and other AWS services.

This immediately tells me:

* EC2 needs to call AWS APIs.
* The question is asking **how EC2 should authenticate**.

It's **not** asking about network access or permissions.

---

# Step 2: How can EC2 talk to AWS?

Suppose your application wants to upload a file to S3.

```python
s3.upload_file(...)
```

AWS asks:

> "Who are you?"

The application must authenticate.

There are several ways.

---

### Option 1 (Bad)

Store Access Key + Secret Key

```
AKIAxxxxxxxx

Secret Key
```

inside the EC2 instance.

This works.

But it's a terrible idea.

Why?

* Keys never expire (unless rotated).
* Anyone who steals them can use your AWS account.
* Rotation becomes difficult.
* Developers accidentally commit them to GitHub.

AWS recommends avoiding this.

---

### Option 2 (Best)

Attach an IAM Role.

```
EC2

↓

IAM Role

↓

Temporary Credentials

↓

AWS
```

No keys stored.

No passwords.

AWS automatically provides temporary credentials.

---

# Step 3: What actually happens?

Imagine you launch:

```
EC2
```

and attach:

```
S3AccessRole
```

Inside EC2:

Your application simply runs:

```python
boto3.client("s3")
```

No credentials.

No secrets.

No keys.

How?

AWS automatically supplies temporary credentials through the **Instance Metadata Service (IMDS)**.

The AWS SDK retrieves them automatically.

As developers, we don't usually see this process.

---

# Step 4: What is an IAM Role?

Think of an IAM Role as a **temporary identity**.

A User is a permanent person.

```
Bilal

↓

IAM User

↓

Long-term credentials
```

A Role is temporary.

```
EC2

↓

Assume Role

↓

Temporary credentials
```

Roles are used by:

* EC2
* Lambda
* ECS
* EKS
* Step Functions
* AWS services

Almost everything in AWS uses roles.

---

# Step 5: Why Option A is wrong

> Add API credentials in the Security Group

This doesn't even make sense.

What is a Security Group?

A firewall.

Example:

Allow:

* HTTP
* HTTPS
* SSH

That's all.

Security Groups do **not** store credentials.

Think:

Security Group = Network

IAM = Identity

Different services.

---

# Step 6: Why Option B is wrong

> Store API credentials on a bastion host

Another trick.

What is a Bastion Host?

```
Internet

↓

Bastion

↓

Private EC2
```

Its job is:

SSH access.

Nothing else.

It is NOT a secret storage service.

---

# Step 7: Why Option D is wrong

This one is interesting.

It says:

> Store long-term credentials in Secrets Manager

Sounds secure.

But read carefully.

It then says:

> Store them on the EC2 instance.

That is exactly what AWS doesn't want.

Secrets Manager is great for:

* Database passwords
* API tokens
* Third-party credentials

But **AWS credentials for EC2 itself should come from an IAM Role**, not be stored as long-term secrets.

---

# Step 8: Architect's View

```
         IAM Role
      (S3ReadWrite)

           │
           │
           ▼

       EC2 Instance

           │
           ▼

        S3 Bucket
```

Application:

```
Upload file

↓

SDK gets temporary credentials

↓

IAM Role authorizes request

↓

S3
```

No passwords.

No access keys.

---

# Step 9: Real DevOps Example

Since you've worked with Terraform and EC2 deployments before, you'll likely see this pattern often.

Terraform:

```hcl
resource "aws_iam_role" "ec2_role" {}
```

Then:

```hcl
resource "aws_iam_instance_profile" "profile" {}
```

Then attach it to EC2.

Your application code contains **no AWS keys**.

That's exactly how production environments are designed.

---

# Step 10: One More Concept (Very Important)

People confuse these:

### IAM User

Used for:

* Humans
* CLI
* Console login

Has:

```
Access Key

Secret Key
```

Usually long-term.

---

### IAM Role

Used for:

* EC2
* Lambda
* ECS
* EKS

Has:

Temporary credentials.

Automatically rotated.

Much more secure.

---

# SAA Exam Keywords

Whenever you see:

* EC2 accessing S3
* EC2 accessing DynamoDB
* EC2 accessing SNS
* EC2 accessing SQS
* EC2 accessing Secrets Manager
* EC2 accessing Parameter Store

Think:

> **IAM Role**

Not access keys.

# SAA Cheat Sheet

| Service             | Authentication Method                                                   |
| ------------------- | ----------------------------------------------------------------------- |
| EC2 → S3            | IAM Role                                                                |
| Lambda → DynamoDB   | IAM Role                                                                |
| ECS → S3            | IAM Task Role                                                           |
| EKS Pod → S3        | IAM Role for Service Account (IRSA)                                     |
| Human using AWS CLI | IAM User (or IAM Identity Center / federated identity in modern setups) |

## One exam tip

Whenever an AWS exam question says:

> "An EC2 instance needs permission to access another AWS service."

Before you even read the options, your first thought should be:

> **Attach an IAM Role to the EC2 instance.**

Around **90% of the time**, that will be the correct answer. The only time it's not is when the question is specifically testing another AWS identity feature, but for standard EC2-to-AWS-service access, **IAM Role is the AWS best practice**.
