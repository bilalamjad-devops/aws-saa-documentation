6. Question

Category: CSAA – Design Resilient Architectures

You are automating the creation of EC2 instances in your VPC. Hence, you wrote a python script to trigger the Amazon EC2 API to request 50 EC2 instances in a single Availability Zone. However, you noticed that after 20 successful requests, subsequent requests failed.

What could be a reason for this issue and how would you resolve it?

- By default, AWS allows you to provision a maximum of 20 instances per region. Select a different region and retry the failed request.
- `There is a vCPU-based On-Demand Instance limit per region which is why subsequent requests failed. Just submit the limit increase form to AWS and retry the failed requests once approved.`
- There was an issue with the Amazon EC2 API. Just resend the requests and these will be provisioned successfully.
- By default, AWS allows you to provision a maximum of 20 instances per Availability Zone. Select a different Availability Zone and retry the failed request.

Let's study both the **exam concept** and the **real-world concept**.

Notice something.

The question **never says**:

* Out of capacity
* InsufficientInstanceCapacity
* Wrong AMI
* Security Group issue

Instead it asks:

Why would AWS stop after a fixed number?

That should make you think:

> **Quota (Limit)**

---

# Step 2: AWS has Service Quotas

AWS doesn't let new accounts create unlimited resources.

Examples:

EC2

Elastic IPs

VPCs

EBS Volumes

IAM Roles

All have limits.

This protects AWS from:

* accidental scripts
* infinite loops
* abuse
* unexpected costs

---

# Step 3: What limit is being tested?

Many beginners think:

> "Maximum EC2 instances = 20"

That used to be closer to older account defaults, but **today AWS primarily uses vCPU-based On-Demand quotas**, not a simple instance-count limit.

Example:

Suppose:

```text
t3.micro = 2 vCPUs
```

If your regional quota is:

```text
32 vCPUs
```

You can launch:

```text
16 t3.micro
```

But if you choose:

```text
c5.4xlarge

16 vCPUs
```

You could launch only:

```text
2 instances
```

before hitting the quota.

So AWS measures:

> **vCPUs**

Not simply:

> Number of EC2 instances.

That's why the correct answer mentions a **vCPU-based On-Demand Instance limit per Region**.

---

# Step 4: Why is it Regional?

Imagine AWS has:

```text
us-east-1

↓

Quota = X vCPUs
```

and

```text
eu-west-1

↓

Another independent quota
```

Limits are managed:

Per Region.

Not globally.

Not per AZ.

---

# Step 5: Why not Availability Zone?

The question purposely says:

> single Availability Zone

Many students think:

"Oh...

Maybe each AZ has a limit."

No.

Quota belongs to:

Region.

Example:

```text
us-east-1

AZ A

AZ B

AZ C
```

All three AZs share the Regional quota.

---

# Step 6: How do you solve it?

AWS says:

Request a quota increase.

Today you typically do this through the **Service Quotas** console (or the EC2 console for some quotas).

AWS reviews it.

If approved:

New limit.

Done.

# Step 11: Real DevOps Example

Suppose you write:

```python
for i in range(100):
    ec2.run_instances(...)
```

The first group launches.

Then AWS returns an error similar to:

```text
VcpuLimitExceeded
```

or another quota-related error depending on the request.

Nothing is wrong with your code.

AWS is protecting your account from exceeding its allowed capacity.

|-----------|--------|

## One note about the wording

This Tutorials Dojo question says the first **20 requests** succeeded and the rest failed. Don't memorize the number **20**.

The important concept is:

- The account hit its **regional vCPU quota**.
- The exact number of instances you can launch depends on the **instance type** (because different instance types have different numbers of vCPUs) and your account's current quota.

For the SAA exam, remember this rule:

> If EC2 launches fail after reaching a predictable threshold and there's no indication of capacity shortages or configuration errors, think **Service Quotas (regional vCPU-based On-Demand limits)** before anything else.
