1. Question

Category: CSAA – Design Resilient Architectures

A company has a cloud architecture composed of Linux and Windows Amazon EC2 instances that process high volumes of financial data 24 hours a day, 7 days a week. To ensure high availability of the systems, the Solutions Architect must create a solution that enables monitoring of memory and disk utilization metrics for all instances.

Which of the following is the most suitable monitoring solution to implement?

- Use Amazon Inspector and install the Inspector agent to all EC2 instances.
- `Install the Amazon CloudWatch agent to all the EC2 instances that gather the memory and disk utilization data. View the custom metrics in the CloudWatch console.`
- Use the default Amazon CloudWatch configuration to EC2 instances where the memory and disk utilization metrics are already available. Install the AWS Systems Manager (SSM) Agent on all the EC2 instances and enable AWS Systems Manager Inventory to track instance configuration and metadata.
- Enable the Enhanced Monitoring option in EC2 and install Amazon CloudWatch agent to all the EC2 instances to be able to view the memory and disk utilization in the CloudWatch dashboard.

# Step 2: What metrics does EC2 give automatically?

When an EC2 instance launches...

CloudWatch automatically receives some metrics.

Think of them as **hardware metrics AWS can observe from outside the VM.**

AWS can see:

✅ CPU Utilization

✅ Network In

✅ Network Out

✅ Disk Read Bytes

✅ Disk Write Bytes

✅ Disk Read Ops

✅ Disk Write Ops

✅ Status Check Failed

---

Notice something?

There is **NO Memory Utilization.**

There is **NO Disk Space Used (%).**

Why?

Because AWS cannot look inside your operating system.

Imagine AWS owns the building.

Your EC2 instance is an apartment.

AWS knows

* electricity usage
* network cables
* CPU hypervisor

But AWS doesn't know

* how much RAM your application is using
* how full C:\ drive is
* Linux swap usage

Those are inside your apartment.

So AWS needs someone inside.

That someone is:

> **CloudWatch Agent**

# Step 4: Why is Option B correct?

> Install CloudWatch Agent

Exactly.

The agent runs inside Linux and Windows.

It collects:

* RAM usage
* Disk utilization
* Swap
* Page file
* Logs

Then sends them as **Custom Metrics** to CloudWatch.

Notice Tutorials Dojo also says:

> View the custom metrics

Why "custom"?

Because AWS isn't generating them.

The agent is.

---

# Step 5: Why Option A is wrong

> Amazon Inspector

Ask yourself:

"What is Inspector?"

Not monitoring.

Inspector = **Security**

Think:

* CVEs
* Vulnerabilities
* Missing patches
* Exposed software
* Security findings

It does NOT monitor memory.

---

Memory monitoring ❌

Security scanning ✅

# Step 6: Why Option C is wrong

This option is tricky.

It says:

> memory and disk metrics are already available

This sentence is false.

That's the trap.

Default CloudWatch provides:

CPU

NOT Memory

NOT Disk Usage

---

It also mentions:

> Systems Manager Inventory

Inventory means exactly what the name says.

Imagine taking inventory in a warehouse.

It tells you:

* OS version

* Installed software

* Applications

* Configuration

It does NOT tell you

RAM = 75%

Disk = 81%

Those are performance metrics.

---

# Step 7: Why Option D is wrong

It mentions

> Enhanced Monitoring

Many students panic because it sounds powerful.

But...

Enhanced Monitoring belongs to:

> **Amazon RDS**

NOT EC2.

Huge exam favorite.

Remember:

EC2 → CloudWatch Agent

RDS → Enhanced Monitoring

# Step 9: Real DevOps Example

You've already worked with EC2, Terraform, monitoring, and production-style deployments, so you'll likely use this in practice as well.

For example, if a Flask application starts slowing down:

Without the CloudWatch Agent:

* CPU = 20%
* Network = Normal

Everything appears healthy.

With the CloudWatch Agent:

* Memory = 97%
* Disk = 98%
* Swap = 4 GB

Now it's obvious the server is running out of resources.

This is why production teams almost always install the CloudWatch Agent on EC2 instances.

# Step 10: SAA Memory Trick

When you see these words, think **CloudWatch Agent**:

* Memory
* RAM
* Disk Space
* Swap
* Page File
* Logs
* Custom Metrics

When you see these services, think of their primary purpose:

* **CloudWatch** → Monitoring
* **CloudWatch Agent** → OS-level metrics and logs
* **Amazon Inspector** → Security and vulnerability assessments
* **AWS Systems Manager Inventory** → Instance inventory and configuration
* **Enhanced Monitoring** → RDS operating system metrics
