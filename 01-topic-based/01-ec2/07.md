7. Question

Category: CSAA – Design Cost-Optimized Architectures

The media company that you are working for has a video transcoding application running on Amazon EC2. Each EC2 instance polls a queue to find out which video should be transcoded, and then runs a transcoding process. If this process is interrupted, the video will be transcoded by another instance based on the queuing system. This application has a large backlog of videos which need to be transcoded. Your manager would like to reduce this backlog by adding more EC2 instances, however, these instances are only needed until the backlog is reduced.

In this scenario, which type of Amazon EC2 instance is the most cost-effective type to use?

- Spot instances
- Reserved instances
- Dedicated instances
- On-demand instances

  This is **one of the most common EC2 pricing questions** on the SAA exam. The exam isn't really testing pricing—it is testing whether you can identify **interruptible vs. non-interruptible workloads**.

There's one sentence in the question that gives away the answer.

---

# Step 1: Read the question

The application:

* Transcodes videos
* Uses a queue
* Multiple EC2 instances

Nothing special yet.

---

Then it says:

> **If this process is interrupted, another instance will continue the work.**

⭐ **This is the key sentence.**

Whenever you see words like:

* interrupted
* restarted
* retry
* checkpoint
* queue
* batch
* fault tolerant

Your brain should immediately think:

> **Spot Instances**

---

# Step 2: Why?

Let's understand the architecture.

```
Videos

↓

Queue

↓

EC2 Instance 1

↓

Transcoding
```

Suppose EC2 dies.

What happens?

The queue still contains the job (or makes it visible again after the visibility timeout, depending on the queue implementation).

Another instance picks it up.

```
Queue

↓

EC2 Instance 2

↓

Continue processing
```

No important data is lost.

This is exactly the kind of workload Spot Instances are designed for.

---

# Step 3: What are Spot Instances?

AWS has unused servers.

Instead of letting them sit idle...

AWS says:

> "I'll rent them very cheaply."

The catch:

AWS can reclaim them.

You receive approximately:

> **2-minute interruption notice**

Then:

Instance may terminate, stop, or hibernate (depending on configuration and support).

---

# Step 4: Why are Spot Instances perfect here?

Question requirements:

✔ Temporary workers

✔ Large backlog

✔ Can tolerate interruption

✔ Wants lower cost

This is literally the textbook definition of a Spot workload.

---

# Step 5: Real-world Example

Suppose Netflix has:

500,000 videos.

Need to encode them.

Can they use Spot?

Absolutely.

If one machine disappears...

Another continues.

No customer notices.

Huge cost savings.

# Step 6: Why Reserved Instances are wrong

Many beginners see:

> "Long workload"

and choose Reserved.

Wrong.

Reserved Instances are for:

**Predictable, long-running capacity where you want a billing discount.**

Example:

Database

Authentication Server

Application Server

These run continuously.

This question says:

> Only needed until backlog is reduced.

Temporary.

Reserved doesn't fit.

Dedicated Instances mean:

* Isolation

Nothing in the question mentions:

Easy elimination.

# Step 11: The Hidden Clue

The question mentions:

> polls a queue

AWS intentionally included that.

Queues make applications resilient.

Imagine using Amazon SQS.

Worker:

```
Receive Message

↓

Process Video

↓

Delete Message
```

If the instance crashes before deleting the message:

The message becomes visible again after the visibility timeout.

Another worker processes it.

Exactly why Spot works well.

# EC2 Pricing Cheat Sheet

| EC2 Type                    | Best For                                  | Can Be Interrupted? | Cost                 |
| --------------------------- | ----------------------------------------- | ------------------- | -------------------- |
| On-Demand                   | Short-term, unpredictable workloads       | ❌                   | High                 |
| Reserved Instances          | Long-running, predictable workloads       | ❌                   | Lower than On-Demand |
| Spot Instances              | Fault-tolerant, interruptible workloads   | ✅                   | Lowest               |
| Dedicated Instances / Hosts | Compliance, licensing, hardware isolation | ❌                   | Highest              |

---

## Exam shortcut

Whenever an SAA question asks you to choose an EC2 purchasing option, look for these clues **before** looking at the prices:

* **Can the workload be interrupted and restarted?** → **Spot**
* **Does it run continuously for months or years?** → **Reserved Instances or Savings Plans**
* **Is it unpredictable or short-lived?** → **On-Demand**
* **Does it require dedicated physical hardware for licensing/compliance?** → **Dedicated Instances/Hosts**

This single decision tree will answer a large percentage of EC2 pricing questions on the exam.


# 🧩 What is a **Backlog**
- In computing or project work, a **backlog** means a **queue of pending tasks** that haven’t been processed yet.  
- In your scenario: the media company has a **large backlog of videos** → meaning many videos are waiting in the queue to be transcoded.  
- Backlog grows when demand (videos submitted) is higher than processing capacity (EC2 instances available).

---

# 🧩 What is **Transcoding**
- **Transcoding** is the process of converting a video file from one format/codec/bitrate to another.  
- Example: converting a raw `.mov` file into compressed `.mp4` for streaming.  
- It involves decoding the original video, then re‑encoding it into the target format.  
- Used for:
  - Making videos compatible with different devices/platforms.  
  - Reducing file size for storage/streaming.  
  - Adjusting resolution/bitrate for performance.
