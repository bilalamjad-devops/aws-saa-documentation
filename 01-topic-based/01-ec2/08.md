8. Question

Category: CSAA – Design High-Performing Architectures

A company has a highly available architecture consisting of an Elastic Load Balancer and multiple Amazon EC2 instances configured with Auto Scaling across three Availability Zones. The company needs to monitor EC2 instances based on a specific metric that is not readily available in Amazon CloudWatch.

Which of the following is a custom metric in CloudWatch that requires manual setup?


- Disk Read activity of an EC2 instance
- CPU Utilization of an EC2 instance
- `Memory Utilization of an EC2 instance`
- Network packets out of an EC2 instance

😂 This question is basically a **repeat** of Question 1. AWS asks this concept **many times** because it's one of the highest-frequency topics in the SAA exam.

The concept is simple:

> **CloudWatch only knows what it can observe from outside the EC2 instance.**
>
> **If you need operating system information, install the CloudWatch Agent.**

---

# Step 1: What is the question asking?

> Which metric **is NOT available by default** in CloudWatch?

Keyword:

> **not readily available**

Meaning:

Needs a **Custom Metric** (typically collected using the CloudWatch Agent).

---

# Step 2: Think like AWS

Imagine AWS owns the physical server.

Your EC2 is like a virtual machine inside that server.

AWS can see:

```text
CPU

Network

Disk I/O
```

because these are controlled by the hypervisor.

But AWS **cannot see inside your operating system**.

It doesn't automatically know:

* RAM usage
* Swap usage
* C:\ drive usage
* `/var` partition usage
* Running processes

For that, something **inside** the instance has to report the information.

That "something" is the **CloudWatch Agent**.

---

# Step 3: Option Analysis

## Option A

> Disk Read Activity

AWS already knows this.

It can measure:

```text
Application

↓

Disk Read

↓

Hypervisor sees it

↓

CloudWatch
```

No agent needed.

❌ Wrong.

---

## Option B

> CPU Utilization

Most common CloudWatch metric.

Automatically available.

No setup.

❌ Wrong.

---

## Option C ✅

> Memory Utilization

AWS cannot see RAM usage inside your operating system by default.

Need:

CloudWatch Agent

↓

Custom Metric

↓

CloudWatch

Exactly correct.

---

## Option D

> Network Packets Out

AWS sees network traffic.

Already available.

No custom metric.

❌ Wrong.

---

# Step 4: The Exam Pattern

Whenever you see:

> Memory

Immediately think:

> **CloudWatch Agent**

No need to even read the other options.

# Step 7: Easy Memory Rule

Imagine you are standing **outside** a house.

You can observe:

* Lights on or off (CPU activity)
* Cars coming and going (Network)
* Deliveries (Disk I/O)

But you **cannot see inside** to know:

* How many people are in the house (Memory)
* How full the closets are (Disk usage)

Someone inside has to tell you.

That's the CloudWatch Agent.

---

# Step 8: Real DevOps Example

Suppose your application crashes.

CloudWatch shows:

```text
CPU = 15%

Network = Normal
```

Looks healthy.

Then you install the CloudWatch Agent.

Now you see:

```text
Memory = 99%

Swap = Full

Disk = 98%
```

Now the problem is obvious.

This is why production environments almost always install the CloudWatch Agent.

---

# Step 9: One Small Update (for real AWS)

The Tutorials Dojo explanation mentions **CloudWatch Monitoring Scripts written in Perl**. Historically, AWS provided those scripts, but today the **recommended approach is to use the CloudWatch Agent** for collecting memory, disk, swap, logs, and other OS-level metrics.

For the SAA exam, remember:

> **Memory Utilization = CloudWatch Agent = Custom Metric**

That's the key concept AWS wants you to know.
