8. Question

Category: CSAA – Design Resilient Architectures

There was an incident in a production environment where user data stored in an Amazon S3 bucket was accidentally deleted by a Junior DevOps Engineer. The issue was escalated to management, and after a few days, an instruction was given to improve the security and protection of AWS resources.

What combination of the following options will protect the S3 objects in the bucket from both accidental deletion and overwriting? (Select TWO.)

- `Enable Versioning`
- Enable S3 Intelligent-Tiering
- `Enable Multi-Factor Authentication Delete`
- Provide access to S3 data strictly through pre-signed URL only
- Disallow S3 Delete using an IAM bucket policy

# First understand the problem

A junior engineer accidentally deleted files.

Management wants protection against:

* ❌ Accidental deletion
* ❌ Accidental overwrite

Notice:

They **don't** want to stop users from working.

They just want recovery.

---

# Answer 1: Enable Versioning ✅

Think of Versioning like **Git for S3**.

Without versioning:

```
report.pdf
```

Someone overwrites it.

```
report.pdf (new)
```

Old file is gone forever.

---

With Versioning:

```
Version 1 → report.pdf
Version 2 → report.pdf
Version 3 → report.pdf
```

Old versions are still stored.

If someone deletes the object:

```
Delete Marker
```

The older versions still exist.

You can restore them.

---

## Versioning protects against

✅ Accidental delete

✅ Accidental overwrite

This is the biggest keyword.

---

# Answer 2: MFA Delete ✅

Imagine someone tries to permanently delete an object.

AWS asks:

> Enter your MFA code.

No MFA?

No permanent deletion.

It also protects changing the bucket's versioning settings.

So:

```
Delete object permanently
↓

Need MFA
```

Extra protection.

---

# Why not Bucket Policy? ❌

Option:

> Deny Delete

Sounds secure.

But think carefully.

The company still needs developers to delete files when necessary.

If you deny Delete:

```
Nobody can delete anything.
```

That wasn't the requirement.

The requirement says:

> Prevent accidental deletion.

Not:

> Disable deletion forever.

---

# Why not Pre-Signed URL? ❌

Pre-signed URLs only allow temporary access.

Example:

```
Download photo.jpg
```

or

```
Upload photo.jpg
```

They do **nothing** to recover deleted objects.

---

# Why not Intelligent-Tiering? ❌

Students often confuse this.

Intelligent-Tiering only saves money.

It moves objects automatically:

```
Frequent
↓

Infrequent
↓

Archive
```

It has **nothing** to do with deletion.

---

# Easy Memory Trick

### Versioning

> Recovery

---

### MFA Delete

> Extra protection before permanent delete

---

### Bucket Policy

> Controls permissions

---

### Intelligent Tiering

> Saves money

---

### Pre-Signed URL

> Temporary access

---

# Exam Cheat Sheet

| Requirement                | Answer                |
| -------------------------- | --------------------- |
| Recover deleted files      | ✅ Versioning          |
| Recover overwritten files  | ✅ Versioning          |
| Prevent permanent deletion | ✅ MFA Delete          |
| Reduce storage cost        | ✅ Intelligent-Tiering |
| Temporary object access    | ✅ Pre-Signed URL      |
| Allow/Deny permissions     | ✅ Bucket Policy       |

## One-line memory rule

> **Delete or overwrite protection → Versioning + MFA Delete.**
