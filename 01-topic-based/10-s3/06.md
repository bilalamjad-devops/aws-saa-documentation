6. Question

Category: CSAA – Design Secure Architectures

An online medical system hosted in AWS stores sensitive Personally Identifiable Information (PII) of the users in an Amazon S3 bucket. Both the master keys and the unencrypted data should never be sent to AWS to comply with the strict compliance and regulatory requirements of the company.

Which S3 encryption technique should the Architect use?

- Use S3 server-side encryption with an AWS KMS key.
- Use S3 server-side encryption with customer provided key.
- `Use S3 client-side encryption with a client-side master key.`
- Use S3 client-side encryption with an AWS KMS key.

This is one of the **most confusing S3 encryption questions**, but there is a very simple trick.

---

# Step 1: Read the keywords

The question says:

* ✅ Sensitive PII
* ✅ Master keys should **never** be sent to AWS
* ✅ Unencrypted data should **never** be sent to AWS

These two lines decide the answer.

---

# Think about where encryption happens

There are only **2 places** encryption can happen.

## Option 1: Server-Side Encryption (SSE)

```text
Laptop
      |
Plain Data
      |
      ▼
AWS S3
Encrypts it
```

AWS receives **plain (unencrypted)** data first.

❌ This violates:

> Unencrypted data should never be sent to AWS.

So eliminate every **Server-Side Encryption** option.

---

## Option 2: Client-Side Encryption

```text
Laptop
Encrypt first
      |
Encrypted Data
      |
      ▼
AWS S3
```

AWS **never** sees the plaintext.

✅ First requirement satisfied.

---

# Now look at the second requirement

It says:

> Master keys should never be sent to AWS.

There are two client-side options.

---

## Client-side with AWS KMS

```text
Your App
      |
Uses AWS KMS Key
      |
AWS involved
```

AWS KMS is still involved.

AWS knows about the key.

❌ Requirement violated.

---

## Client-side with your own Master Key

```text
Your Computer

Your Own Key
      │
Encrypt
      │
Encrypted File
      │
Upload
      ▼
S3
```

AWS never receives:

* plaintext
* master key

✅ Perfect.

---

# Correct Answer

✅ **Use S3 client-side encryption with a client-side master key.**

---

# Why the others are wrong

### ❌ SSE-KMS

AWS encrypts after upload.

AWS also manages KMS.

Plaintext reaches AWS.

---

### ❌ SSE-C

Many students confuse this.

You provide your own key...

BUT

Encryption still happens **inside S3**.

Flow:

```text
Your PC

Plain file
      │
      ▼
S3
Encrypts using your key
```

Plaintext still reaches AWS.

❌

---

### ❌ Client-side with AWS KMS

Encryption happens locally.

Good.

But key management still involves AWS KMS.

Question says

> master keys should NEVER be sent to AWS

❌

---

# Easy Memory Table

| Method                            | Where is data encrypted? | Who has the key?                            |
| --------------------------------- | ------------------------ | ------------------------------------------- |
| SSE-S3                            | AWS                      | AWS                                         |
| SSE-KMS                           | AWS                      | AWS KMS                                     |
| SSE-C                             | AWS                      | You provide the key to AWS for each request |
| Client-side + KMS                 | Your computer            | AWS KMS                                     |
| **Client-side + Your Master Key** | **Your computer**        | **Only you** ✅                              |

---

# 10-Second Exam Trick

If the question says:

> **AWS should never see the plaintext**

👉 **Client-side encryption**

If it also says:

> **AWS should never see your master key**

👉 **Client-side encryption with your own master key**

---

## One-line memory

> **Server-side = AWS encrypts.**
> **Client-side = You encrypt before upload.**
> **Maximum security = Client-side encryption + your own master key.**
