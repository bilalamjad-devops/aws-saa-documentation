7. Question

Category: CSAA – Design Resilient Architectures

A web application hosted in an Auto Scaling group of EC2 instances in AWS. The application receives a burst of traffic every morning, and a lot of users are complaining about request timeouts. The EC2 instance takes 1 minute to boot up before it can respond to user requests. The cloud architecture must be redesigned to better respond to the changing traffic of the application.

How should the Solutions Architect redesign the architecture?

- Create a CloudFront distribution and set the EC2 instance as the origin.
- `Create a step scaling policy and configure an instance warm-up time condition.`
- Create a Network Load Balancer with slow-start mode.
- Create a new launch template and upgrade the size of the instance.


Excellent. This question introduces another ASG concept that many candidates confuse:

> **Instance Warm-up**

This is **different from Cooldown**, even though both involve waiting.

Let's understand the difference first.

---

# Step 1: Read the scenario

Important clues:

* Every morning → Burst of traffic
* Users experience timeouts
* EC2 takes **1 minute to boot**

That last sentence is the biggest clue.

AWS is telling you:

> **The problem is not scaling.**
>
> **The problem is that new instances aren't ready yet.**

---

# Step 2: What happens?

Imagine this timeline.

### 9:00 AM

```text
Traffic suddenly increases
```

↓

ASG launches 2 EC2 instances.

Great.

---

### 9:00:10

The new EC2 is still:

* Booting
* Running user data
* Starting the application
* Registering with the ALB

It **cannot** serve traffic yet.

---

### But ASG looks at metrics...

Maybe CPU is still high.

Maybe requests are timing out.

AWS needs to know:

> "Should I evaluate this new instance immediately?"

Answer:

No.

Wait until it's actually ready.

That's what **Instance Warm-up** does.

---

# Step 3: What is Instance Warm-up?

Warm-up tells Auto Scaling:

> **Ignore this new instance for scaling decisions until it has had enough time to become fully operational.**

Example:

```text
Launch EC2

↓

Boot

↓

Application Starts

↓

Health Checks

↓

Warm-up Complete

↓

Now include it in scaling metrics
```

---

# Step 4: Why?

Without Warm-up

```text
Launch EC2

↓

Immediately measure metrics

↓

Metrics still high

↓

Launch more EC2

↓

Repeat
```

Result:

Over-scaling.

---

With Warm-up

```text
Launch EC2

↓

Wait 60 seconds

↓

Instance becomes healthy

↓

Now evaluate metrics again
```

Much better.

---

# Step 5: Why Step Scaling?

The answer says:

> Step Scaling + Warm-up

Because Step Scaling supports configuring an **instance warm-up period** for newly launched instances.

The key part of the answer is really the **warm-up**, which addresses the boot delay described in the question.

---

# Step 6: Cooldown vs Warm-up (Very Important)

This is where many students get confused.

## Cooldown

Question:

Should ASG perform another scaling action yet?

Cooldown says:

```text
Wait before making another scaling decision.
```

It's about **the Auto Scaling Group**.

---

## Warm-up

Question:

Should this **new EC2 instance** already count in scaling metrics?

Warm-up says:

```text
Wait until the instance is fully ready.
```

It's about **the newly launched instance**.

---

### Easy Comparison

| Feature      | Purpose                                                                                              |
| ------------ | ---------------------------------------------------------------------------------------------------- |
| **Cooldown** | Prevent repeated scaling actions too quickly                                                         |
| **Warm-up**  | Give new EC2 instances time to become fully operational before they're included in scaling decisions |

---

# Step 7: Option Analysis

### Option A

CloudFront

CloudFront caches content.

The problem isn't latency.

It's that EC2 isn't ready.

❌ Wrong.

---

### Option B ✅

Step Scaling

*

Warm-up

Exactly addresses:

EC2 needs one minute before it can serve traffic.

---

### Option C

Network Load Balancer

It doesn't support **slow start mode**.

Also, slow start is a feature of the **Application Load Balancer**, not NLB.

❌ Wrong.

---

### Option D

Bigger EC2

A larger instance might improve application performance, but it doesn't guarantee a significantly faster boot process.

The question specifically tells you the issue is **boot time**.

❌ Wrong.

---

# Step 9: Memory Trick

Imagine hiring a new employee.

First day.

Do you judge their productivity after:

2 minutes?

No.

Give them training first.

Then evaluate.

That's Warm-up.

---

Cooldown is different.

It's like the manager saying:

> "We already hired people.
> Let's wait and see if that's enough before hiring more."

# Step 10: Exam Keywords

Whenever you read:

* Boot time
* Startup time
* Initialization
* User Data
* Application startup
* Instance not ready

Think:

> **Instance Warm-up**

---

Whenever you read:

* Prevent repeated scaling
* Wait after scaling
* Scaling happens too often

Think:

> **Cooldown**

# Updated ASG Cheat Sheet

| Concept               | Purpose                                                                           |
| --------------------- | --------------------------------------------------------------------------------- |
| **Scheduled Scaling** | Scale at a specific time                                                          |
| **Simple Scaling**    | One threshold → One adjustment                                                    |
| **Step Scaling**      | Multiple thresholds → Multiple adjustments                                        |
| **Target Tracking**   | Maintain target metric                                                            |
| **Cooldown**          | Pause before another scaling action                                               |
| **Warm-up**           | Wait for new EC2 instances to become ready before using them in scaling decisions |

---

# ⭐ Bilal's Golden Rule #17

If the question mentions:

* **EC2 boot time**
* **Startup delay**
* **Initialization**
* **User data execution**
* **New instances not immediately ready**

Think:

> **Instance Warm-up**

If the question mentions:

* **Repeated scaling**
* **Too many scaling actions**
* **Wait before scaling again**

Think:

> **Cooldown**

---

## One small clarification

The Tutorials Dojo answer specifically pairs **Step Scaling** with **instance warm-up**. In modern AWS, **instance warm-up** is also commonly used with **Target Tracking Scaling**. For the exam, focus on the clue the question gives:

* If it emphasizes **boot/initialization time**, the key concept being tested is **instance warm-up**, regardless of the specific scaling policy mentioned.
