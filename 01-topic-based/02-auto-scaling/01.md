
1. Question

Category: CSAA – Design Resilient Architectures

A major TV network has a web application running on eight Amazon T3 EC2 instances behind an application load balancer. The number of requests that the application processes are consistent and do not experience spikes. A Solutions Architect must configure an Auto Scaling group for the instances to ensure that the application is running at all times.

Which of the following options can satisfy the given requirements?

- Deploy four EC2 instances with Auto Scaling in one region and four in another region behind an Amazon Elastic Load Balancer.
- `Deploy four EC2 instances with Auto Scaling in one Availability Zone and four in another availability zone in the same region behind an Amazon Elastic Load Balancer.`
- Deploy two EC2 instances with Auto Scaling in four regions behind an Amazon Elastic Load Balancer.
- Deploy eight EC2 instances with Auto Scaling in one Availability Zone behind an Amazon Elastic Load Balancer.

Great! Welcome to the **Auto Scaling Group (ASG)** section. This first question teaches one of the **most fundamental AWS architecture principles**:

> **For High Availability, use Multiple Availability Zones—not Multiple Regions.**

Let's break it down.

---

# Step 1: Read the requirements

The application has:

* ✅ 8 EC2 instances
* ✅ Application Load Balancer (ALB)
* ✅ Requests are **consistent** (no spikes)
* ✅ Must **always be available**

Notice what is **not** mentioned:

* Cost optimization
* Global users
* Disaster recovery
* Multi-region

The only goal is:

> **High Availability**

---

# Step 2: What is Auto Scaling actually doing here?

Many beginners think:

> Auto Scaling = Scaling Up

Not always.

Auto Scaling has **three jobs**:

1. **Maintain desired capacity** ✅
2. **Replace unhealthy instances** ✅
3. **Scale in/out when needed** ✅

In this question, because traffic is **constant**, Auto Scaling is mainly being used for **self-healing**, not scaling.

---

# Step 3: Why Option B is correct

Option B says:

> 4 EC2s in AZ-A

> 4 EC2s in AZ-B

Behind one ALB.

Architecture:

```text
                Internet
                    │
                 ALB
          ┌─────────┴─────────┐
          │                   │
        AZ-A               AZ-B
      EC2 EC2 EC2 EC2   EC2 EC2 EC2 EC2
          \               /
           \             /
          Auto Scaling Group
```

Suppose AZ-A fails.

Remaining:

```text
                Internet
                    │
                 ALB
                    │
                 AZ-B
          EC2 EC2 EC2 EC2
```

The application still works.

Then Auto Scaling launches more instances in AZ-B until the desired capacity is restored (subject to capacity and configuration).

Exactly what AWS wants.

---

# Step 4: Why doesn't AWS say "8 instances in each AZ"?

Good observation.

You may remember the previous question where we needed **4 instances minimum** to survive an AZ failure.

Why is this question different?

Because look carefully.

Previous question said:

> **Mission-critical**

and

> **Must always have at least 2 running**

This question only says:

> Ensure the application is running.

It **does not** say:

* Maintain full capacity
* Zero performance degradation
* Fault tolerant

Some degradation is acceptable.

Availability is the priority.

This is an important distinction in SAA questions.

---

# Step 5: Why Option A is wrong

It says:

> One Region

and

> Another Region

Sounds even better, right?

No.

Here's the trick.

Question says:

Behind **an** Application Load Balancer.

An **Application Load Balancer is regional**.

Example:

```
ALB (us-east-1)
```

It can distribute traffic across:

* us-east-1a
* us-east-1b
* us-east-1c

But **not** to:

```
eu-west-1
```

ALB cannot span Regions.

For multi-Region architectures, you would use services such as **Amazon Route 53**, **AWS Global Accelerator**, or **CloudFront** to direct traffic between Regions—not a single ALB.

---

# Step 6: Why Option C is wrong

It says:

> Four Regions

Same mistake.

One ALB cannot load balance across four Regions.

Immediate elimination.

---

# Step 7: Why Option D is wrong

Everything is in one AZ.

```
AZ-A

EC2 x8
```

Looks fine...

Until:

```
AZ-A ❌
```

Everything disappears.

Single point of failure.

Never do this for production.

---

# Step 8: Important AWS Concept

## Region

A Region is a geographic area.

Example:

```
us-east-1
```

Inside that Region are multiple Availability Zones.

---

## Availability Zone

```
Region

├── AZ-A
├── AZ-B
├── AZ-C
```

Each AZ is:

* Independent power
* Independent networking
* Independent cooling

AWS recommends deploying applications across multiple AZs for High Availability.

---

# Step 9: Auto Scaling + ALB

This is the most common AWS architecture you'll see in the SAA exam.

```
Internet
    │
   ALB
    │
──────────────
│            │
AZ-A       AZ-B
EC2        EC2
│            │
└──── ASG ───┘
```

The ALB sends traffic only to healthy instances.

The ASG ensures the required number of instances exist.

Together they provide:

* High Availability
* Self-Healing
* Elasticity

---

# Step 10: T3 Instance Mention

The explanation mentions:

> T3 Burstable Performance

Don't overthink it.

T3 instances accumulate **CPU credits**.

When CPU demand suddenly increases...

They can temporarily burst above their baseline performance.

For this question, that's just extra context.

The real topic being tested is:

> Multi-AZ Auto Scaling.

---

# Step 11: SAA Exam Tricks

Whenever you see:

> Highly Available Web Application

Think:

```
ALB

↓

Multiple AZs

↓

Auto Scaling Group
```

---

Whenever you see:

> One Availability Zone

Think:

❌ Single Point of Failure

---

Whenever you see:

> One ALB

Think:

Regional.

Not cross-region.

---

# Memory Trick

Imagine a shopping mall.

One Region:

```
City
```

Availability Zones:

```
Mall A

Mall B

Mall C
```

Your store opens in:

Mall A

and

Mall B.

If Mall A closes because of a power outage...

Customers still shop at Mall B.

That's High Availability.

Opening stores in another **country** (another Region) isn't required for this scenario and wouldn't be handled by a single ALB.

---

# ASG Cheat Sheet #1

| Requirement            | AWS Solution       |
| ---------------------- | ------------------ |
| High Availability      | Multiple AZs       |
| Load Balancing         | ALB                |
| Replace failed EC2     | Auto Scaling Group |
| One ALB across Regions | ❌ Not possible     |
| One ALB across AZs     | ✅ Yes              |
| Single AZ production   | ❌ Avoid            |

---

## ⭐ One New Golden Rule (Add this to your notebook)

> **Application Load Balancer is a Regional service.**
>
> * ✅ It can distribute traffic across **multiple Availability Zones within the same Region**.
> * ❌ It cannot distribute traffic across **multiple AWS Regions**.

This rule alone helps eliminate many SAA exam options.
