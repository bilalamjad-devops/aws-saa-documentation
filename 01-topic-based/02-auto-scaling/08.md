8. Question

Category: CSAA – Design High-Performing Architectures

A commercial bank has designed its next-generation online banking platform to use a distributed system architecture. The Solutions Architect was tasked to ensure that the architecture is highly scalable, cost-effective, and leverages Amazon EC2 for compute resources.

Which of the following will provide the most suitable solution for this scenario?

- Launch multiple EC2 instances and use Elastic Load Balancing to distribute traffic across the instances, which will host application services, while Amazon SNS will act as a highly-scalable buffer for messages as they travel between distributed applications.
- Launch multiple EC2 instances behind an Application Load Balancer to host your application services, and an AWS Step Functions workflow which will act as a highly scalable buffer that stores messages as they travel between distributed applications.
- Launch multiple On-Demand EC2 instances to host your application services and an Amazon SQS queue which will act as a highly-scalable buffer that stores messages as they travel between distributed applications.
- `Launch an Auto-Scaling group of EC2 instances to host your application services and an Amazon SQS queue. Include an Auto Scaling trigger to watch the SQS queue size which will either scale in or scale out the number of EC2 instances based on the queue.`

Excellent. This question is testing **three major SAA concepts at once**:

1. **Auto Scaling**
2. **Amazon SQS**
3. **Distributed System Design**

This is a very common AWS architecture pattern.

# Step 2: What problem are we solving?

Imagine this banking application.

Thousands of customers click:

```text
Transfer Money
```

at the same time.

Without a queue:

```text
Users

↓

EC2

↓

Database
```

If 20,000 users arrive together,

EC2 becomes overloaded.

Requests start failing.

---

# Step 3: Introduce SQS

Instead of sending requests directly to EC2:

```text
Users

↓

SQS Queue

↓

EC2 Workers

↓

Database
```

Now customers don't overload the servers.

Their requests wait safely in the queue.

EC2 instances process them one by one (or in parallel).

This is called **decoupling**.

---

# Step 4: Why SQS?

Imagine a bank receives:

100,000 requests.

But EC2 can only process:

10,000 per minute.

Without SQS

```text
100,000 requests

↓

EC2

↓

Crash
```

---

With SQS

```text
100,000 requests

↓

Queue

↓

EC2 processes

↓

Database
```

No requests are lost.

The queue absorbs the traffic spike.

---

# Step 5: Auto Scaling + SQS

Now imagine the queue starts growing.

```text
Queue

500 messages

↓

1000 messages

↓

5000 messages
```

AWS can monitor:

> **ApproximateNumberOfMessagesVisible**

If the queue becomes large,

ASG launches more EC2 workers.

Architecture:

```text
Users

↓

SQS

↓

CloudWatch

↓

Auto Scaling

↓

More EC2 Workers
```

When the queue becomes empty,

ASG removes instances.

Very cost effective.

# Step 6: Why this is powerful

Waste of money.

With Auto Scaling

# Step 8: Real Architecture

```text
             Users
                │
                ▼
        Web Application
                │
                ▼
          Amazon SQS
                │
     Queue Length Metric
                │
                ▼
         CloudWatch Alarm
                │
                ▼
       Auto Scaling Group
                │
                ▼
         EC2 Worker Nodes
                │
                ▼
            Database
```

This is a classic **producer-consumer** architecture:

* **Producer** = Web application sending messages to SQS.
* **Consumer** = EC2 workers reading and processing messages.

---

# Step 9: SQS vs SNS vs Step Functions

This is one of the most important comparison tables in SAA.

| Service            | Purpose                                                    |
| ------------------ | ---------------------------------------------------------- |
| **SQS**            | Store messages until consumers process them (queue/buffer) |
| **SNS**            | Instantly push notifications to subscribers (pub/sub)      |
| **Step Functions** | Coordinate multi-step workflows between AWS services       |

---
