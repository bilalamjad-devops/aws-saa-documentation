2. Question

Category: CSAA – Design High-Performing Architectures

A tech company has a CRM application hosted on an Auto Scaling group of On-Demand EC2 instances with different instance types and sizes. The application is extensively used during office hours from 9 in the morning to 5 in the afternoon. Their users are complaining that the performance of the application is slow during the start of the day but then works normally after a couple of hours.

Which of the following is the MOST operationally efficient solution to implement to ensure the application works properly at the beginning of the day?

- Configure a Dynamic scaling policy for the Auto Scaling group to launch new instances based on the CPU utilization.
- Configure a Predictive scaling policy for the Auto Scaling group to automatically adjust the number of Amazon EC2 instances
- `Configure a Scheduled scaling policy for the Auto Scaling group to launch new instances before the start of the day.`
- Configure a Dynamic scaling policy for the Auto Scaling group to launch new instances based on the Memory utilization.

Excellent. This question introduces the **different ASG scaling policies**, and this is one of the most frequently tested ASG topics.

The entire question can be solved by recognizing **one sentence**.

---

# Step 1: Read the scenario

The application is busy:

> **9 AM to 5 PM**

This is the most important information.

Notice:

* Same time every day.
* Completely predictable.

---

Then:

> Users complain at the **start of the day**.

Why?

Because the extra EC2 instances are **not ready yet**.

EC2 instances take time to:

* Launch
* Boot
* Run user data
* Register with the ALB
* Pass health checks

That can take a few minutes.

# Step 3: Correct Solution

AWS says:

Instead of reacting...

Prepare in advance.

At:

```text
8:45 AM
```

Launch extra instances.

So:

```text
9:00 AM

Everything already running.
```

This is called:

> **Scheduled Scaling**

# Step 4: Why Scheduled Scaling?

Scheduled Scaling is used when traffic is:

Predictable.

Examples:

Every morning

Every Friday

Every month

Black Friday

Office hours

School hours

Payroll day

If you already know **when** traffic increases, don't wait for CloudWatch alarms.

Schedule it.

---

# Step 5: Why Dynamic Scaling is wrong

Dynamic Scaling works like this:

```text
CPU

↓

Crosses threshold

↓

CloudWatch Alarm

↓

ASG launches EC2

↓

Wait
```

It reacts **after** the load increases.

That's too late for this scenario.

---

# Step 6: CPU vs Memory

The options mention:

CPU

and

Memory.

Both are Dynamic Scaling.

Both react **after** the problem starts.

So both are wrong.

---

# Step 7: Why Predictive Scaling is wrong

This is a newer feature.

Predictive Scaling uses:

Machine Learning

↓

Historical traffic

↓

Forecasts future demand

↓

Launches instances automatically

Sounds perfect.

Why isn't it correct?

Read the question carefully.

It says:

> Different instance types and sizes.

Predictive Scaling works best with **homogeneous** Auto Scaling groups, where instance capacity is consistent. With mixed instance types and sizes, forecasting and capacity calculations become less accurate.

Also, the traffic pattern here is **simple and known** (9 AM every weekday). A straightforward Scheduled Scaling policy is the most operationally efficient solution.

# ASG Scaling Cheat Sheet

| Scaling Type           | Trigger              | Best Use Case                            |
| ---------------------- | -------------------- | ---------------------------------------- |
| **Scheduled Scaling**  | Time                 | Predictable traffic (e.g., office hours) |
| **Dynamic Scaling**    | CloudWatch metrics   | Unpredictable traffic                    |
| **Predictive Scaling** | Historical data + ML | Forecasting future demand                |

---

# ⭐ Bilal's Golden Rule #12

When solving ASG questions, ask yourself:

**Do I know *when* traffic will increase?**

* **Yes** → **Scheduled Scaling**
* **No, but I can measure it (CPU, requests, queue length)** → **Dynamic Scaling**
* **I want AWS to learn the pattern and forecast demand automatically** → **Predictive Scaling**

---

## One small note for the real world

AWS **Predictive Scaling** is a valid and useful feature, but for the SAA exam, if the workload has a **simple, fixed schedule** like "every weekday from 9 AM to 5 PM," **Scheduled Scaling** is usually considered the most operationally efficient and easiest-to-manage solution. That's the distinction this question is testing.
