6. Question

Category: CSAA – Design High-Performing Architectures

An application is hosted in an Auto Scaling group of EC2 instances. To improve the monitoring process, you have to configure the current capacity to increase or decrease based on a set of scaling adjustments. This should be done by specifying the scaling metrics and threshold values for the CloudWatch alarms that trigger the scaling process.

Which of the following is the most suitable type of scaling policy that you should use?

- Target tracking scaling
- `Step scaling`
- Scheduled Scaling
- Simple scaling

Excellent! This is probably the **most confusing ASG topic** because there are **4 scaling policies**, and they sound almost identical.

The trick is to understand **how each one makes scaling decisions**.

---

# Step 1: Read the keywords

The question says:

> **increase or decrease based on a set of scaling adjustments**

That phrase is the giveaway.

AWS documentation uses almost the exact same wording.

Whenever you see:

> **set of scaling adjustments**

Think:

> ✅ **Step Scaling**

---

# Step 2: Let's understand all 4 scaling policies

Think of them as four different managers.

---

# 1. Simple Scaling

Very basic.

Rule:

```text
CPU > 70%

↓

Add 1 EC2
```

Only one action.

Always the same.

No matter if CPU is:

72%

or

99%

Still:

```text
+1 Instance
```

Very simple.

---

# 2. Step Scaling ✅

Now AWS becomes smarter.

Instead of one rule...

You create multiple steps.

Example:

```text
CPU > 60%

↓

+1 EC2
```

---

```text
CPU > 75%

↓

+2 EC2
```

---

```text
CPU > 90%

↓

+4 EC2
```

Notice:

Different actions depending on how serious the problem is.

This is why it's called:

**Step Scaling**

Each threshold is another "step."

---

# Step 3: Visual

Imagine stairs.

```text
CPU

60%

↓

+1

──────────

75%

↓

+2

──────────

90%

↓

+4
```

Each stair has its own scaling action.

---

# 4. Target Tracking

This one works differently.

Instead of telling AWS:

"When CPU reaches X..."

You tell AWS:

> Keep CPU around **50%**

That's all.

AWS automatically decides:

* When to scale
* How much to scale

Exactly like a thermostat.

---

Example:

You say:

```text
Target CPU

50%
```

Current:

```text
CPU = 80%
```

AWS adds instances.

Later:

```text
CPU = 45%
```

AWS removes instances.

You don't define the exact scaling adjustments.

AWS does.

---

# Why thermostat?

Suppose your room temperature is:

30°C.

Thermostat target:

22°C.

You don't tell it:

Turn fan speed 3.

Turn fan speed 5.

It figures that out automatically.

Target Tracking works the same way.

---

# 5. Scheduled Scaling

Traffic known in advance.

Example:

```text
Every day

9 AM

↓

Launch 4 EC2
```

No CloudWatch alarms needed.

---

# Step 6: Why Step Scaling is correct

Look at the wording.

Question says:

> **set of scaling adjustments**

Not:

Target value.

Not:

Single adjustment.

Not:

Schedule.

Only Step Scaling matches.

---

# Step 7: Why Target Tracking is wrong

Many students pick this.

Because it sounds modern.

But Target Tracking only asks for:

Target metric.

Example:

Keep CPU at:

```text
50%
```

AWS handles the rest.

You don't define multiple adjustment levels.

---

# Step 8: Why Simple Scaling is wrong

Simple Scaling only has:

One rule.

Example:

```text
CPU > 70%

↓

+1
```

Not:

```text
70%

↓

+1

80%

↓

+2

90%

↓

+4
```

---

# Step 9: Why Scheduled is wrong

Scheduled doesn't care about CPU.

Only time.

Example:

```text
Monday

8 AM

↓

Launch
```


because you've configured larger adjustments for larger breaches.

# Step 11: Exam Cheat Table

| Policy                | You Configure                          | AWS Does                      |
| --------------------- | -------------------------------------- | ----------------------------- |
| **Simple Scaling**    | One threshold → One action             | Executes it                   |
| **Step Scaling**      | Multiple thresholds → Multiple actions | Executes matching step        |
| **Target Tracking**   | Target value (e.g., CPU 50%)           | Decides scaling automatically |
| **Scheduled Scaling** | Time                                   | Executes at that time         |

# Updated ASG Cheat Sheet

| Scaling Policy      | Trigger                               | Best Use                                     |
| ------------------- | ------------------------------------- | -------------------------------------------- |
| **Scheduled**       | Time                                  | Predictable traffic                          |
| **Simple**          | One CloudWatch alarm → One adjustment | Basic scaling (older approach)               |
| **Step**            | Alarm + multiple adjustment levels    | Different scaling based on breach size       |
| **Target Tracking** | Maintain target metric                | Recommended for utilization metrics like CPU |

---

# ⭐ Bilal's Golden Rule #16

When you see an ASG scaling policy question, look for these keywords:

* **"Maintain CPU at 50%"** → **Target Tracking**
* **"Set of scaling adjustments" / "Different actions based on breach size"** → **Step Scaling**
* **"Single scaling adjustment"** → **Simple Scaling**
* **"Every day at 9 AM" / "Known schedule"** → **Scheduled Scaling**

---

## One exam tip

AWS now generally recommends **Target Tracking Scaling** for common utilization metrics (such as average CPU utilization) because it's simpler to manage. However, if the question explicitly mentions **multiple scaling adjustments**, **alarm breach size**, or **step adjustments**, the correct answer is **Step Scaling**. Those phrases are direct clues from the AWS documentation.
