1. Question

Category: CSAA – Design High-Performing Architectures

A popular augmented reality (AR) mobile game is heavily using a RESTful API which is hosted in AWS. The API uses Amazon API Gateway and a DynamoDB table with a preconfigured read and write capacity. Based on your systems monitoring, the DynamoDB table begins to throttle requests during high peak loads which causes the slow performance of the game.

Which of the following can you do to improve the performance of your app?

- Integrate an Application Load Balancer with your DynamoDB table.
- `Use DynamoDB Auto Scaling`
- Add the DynamoDB table to an Auto Scaling Group.
- Create an SQS queue in front of the DynamoDB table.

Perfect! Welcome to **DynamoDB**. Before solving the question, let's build the foundation because many students struggle with DynamoDB simply because they think of it like MySQL or PostgreSQL.

---

# Step 1: What is DynamoDB?

DynamoDB is AWS's **fully managed NoSQL database**.

That means:

* ✅ Serverless
* ✅ Highly Available
* ✅ Multi-AZ by default
* ✅ Extremely fast (single-digit millisecond latency)
* ✅ Can handle millions of requests

Unlike MySQL:

You **don't manage servers**.

AWS manages everything.

# Step 3: What is Throttling?

Imagine a restaurant.

Chef can cook:

```text
100 burgers/hour
```

Suddenly,

Customers order:

```text
300 burgers/hour
```

Restaurant says:

> I can't process everything.

Some customers wait.

Some orders are rejected.

This is exactly what DynamoDB throttling is.

---

# Step 4: Why did throttling happen?

The question says:

> **Preconfigured read and write capacity**

That means the table is using:

**Provisioned Capacity Mode**.

Example:

```text
Read Capacity = 100

Write Capacity = 50
```

Now traffic becomes:

```text
Reads = 250

Writes = 100
```

AWS starts throttling.

---

# Step 5: Solution?

Increase capacity.

But manually changing it every hour is a bad idea.

AWS has a feature:

> **DynamoDB Auto Scaling**

---

Think of it exactly like EC2 Auto Scaling.

Instead of scaling EC2,

It scales:

and

Write Capacity Units (WCUs)

# Step 8: Option Analysis

## Option A

Application Load Balancer

Wrong.

ALB distributes traffic between:

* EC2
* ECS
* Lambda
* IP targets

It cannot sit in front of DynamoDB.

❌ Wrong.

---

## Option B ✅

DynamoDB Auto Scaling

Perfect.

Automatically increases:

* Read Capacity
* Write Capacity

No throttling.

---

## Option C

Put DynamoDB inside an Auto Scaling Group.

Impossible.

ASGs only manage compute resources like EC2.

DynamoDB is a managed database service.

❌ Wrong.

---

## Option D

SQS

SQS stores messages.

It doesn't increase DynamoDB's throughput.

It may help decouple producers and consumers in some architectures, but it doesn't solve the root problem described here: **the table's provisioned capacity is too low**.

❌ Wrong.

# Step 9: DynamoDB Capacity Modes

This is one of the highest-yield DynamoDB topics.

There are **two capacity modes**.

### 1. Provisioned Capacity

You decide:

```text
Reads = 100

Writes = 50
```

Cheaper for predictable workloads.

Can use Auto Scaling.

---

### 2. On-Demand Capacity

AWS decides everything.

Unlimited scaling.

Perfect for:

Unpredictable traffic.

Costs more per request.

---

Memory Trick

Provisioned

↓

"I know my traffic."

On-Demand

↓

"I have no idea."

---

# Step 10: When does throttling happen?

Mostly in:

Provisioned mode.

Because capacity is fixed unless Auto Scaling adjusts it.

On-Demand mode automatically scales with demand, so throttling due to insufficient configured capacity is much less common (though service quotas and partition limits can still matter in extreme cases).

# ⭐ Bilal's Golden Rule #19

When a DynamoDB question mentions:

* **Provisioned capacity**
* **Read/Write Capacity Units (RCUs/WCUs)**
* **Throttling during traffic spikes**

Think:

> **Enable DynamoDB Auto Scaling** (or consider On-Demand capacity if the workload is highly unpredictable).

---

## One important SAA note

There are **two different "Auto Scaling" services** you'll encounter:

* **EC2 Auto Scaling** → Scales **EC2 instances**.
* **Application Auto Scaling** → Scales resources like **DynamoDB provisioned capacity**, Aurora Replicas, ECS services, and more.

A common exam trap is using **EC2 Auto Scaling** for a managed service like DynamoDB. That's never correct.
