# Step 1: What is DynamoDB?

DynamoDB is AWS's **fully managed NoSQL database**.

That means:

* ✅ Serverless
* ✅ Highly Available
* ✅ Multi-AZ by default
* ✅ Extremely fast (single-digit millisecond latency)
* ✅ Can handle millions of requests

Unlike MySQL:

You **don't manage servers**.

AWS manages everything.

# Step 9: DynamoDB Capacity Modes

This is one of the highest-yield DynamoDB topics.

There are **two capacity modes**.

### 1. Provisioned Capacity

You decide:

```text
Reads = 100

Writes = 50
```

Cheaper for predictable workloads.

Can use Auto Scaling.

---

### 2. On-Demand Capacity

AWS decides everything.

Unlimited scaling.

Perfect for:

Unpredictable traffic.

Costs more per request.

---

Memory Trick

---

# Step 10: When does throttling happen?

Mostly in:

Provisioned mode.

Because capacity is fixed unless Auto Scaling adjusts it.

On-Demand mode automatically scales with demand, so throttling due to insufficient configured capacity is much less common (though service quotas and partition limits can still matter in extreme cases).

## One important SAA note

There are **two different "Auto Scaling" services** you'll encounter:

* **EC2 Auto Scaling** → Scales **EC2 instances**.
* **Application Auto Scaling** → Scales resources like **DynamoDB provisioned capacity**, Aurora Replicas, ECS services, and more.

A common exam trap is using **EC2 Auto Scaling** for a managed service like DynamoDB. That's never correct.

🎯 **Excellent!** You're about 80% there. Your answer is correct:

> "We should enable DynamoDB Auto Scaling so it can handle the increased requests."

Now let's answer your excellent follow-up question.

> **Do we also specify min and max like EC2 Auto Scaling?**

**Yes!** But instead of specifying the number of EC2 instances, you specify the **minimum and maximum Read/Write Capacity Units**.

---

# EC2 Auto Scaling

You configure:

```text
Minimum Instances = 2

Desired = 4

Maximum = 10
```

AWS launches or terminates EC2 instances.

---

# DynamoDB Auto Scaling

You configure something like:

```text
Read Capacity

Minimum = 100 RCU

Maximum = 1000 RCU

Target Utilization = 70%
```

And similarly for writes:

```text
Write Capacity

Minimum = 50 WCU

Maximum = 500 WCU

Target Utilization = 70%
```

Notice the difference?

EC2 Auto Scaling manages **instances**.

DynamoDB Auto Scaling manages **capacity units**.

---

## What is Target Utilization?

This is another important concept.

Suppose you set:

```text
Maximum Read Capacity = 1000 RCU

Target Utilization = 70%
```

AWS tries to keep your table around **70% utilized**.

Example:

Current capacity:

```text
100 RCU
```

Traffic uses:

```text
95 RCU
```

Utilization:

```text
95%
```

That's much higher than the 70% target.

AWS says:

> "I should increase the provisioned capacity."

So it might increase it to:

```text
150 RCU
```

Now:

```text
95 / 150 ≈ 63%
```

Much closer to the target.

## One More Important Thing for the SAA Exam

There are **two capacity modes** for DynamoDB:

### 1. Provisioned Capacity

* You define RCUs/WCUs.
* You **can enable Auto Scaling**.
* Best for predictable workloads.

### 2. On-Demand Capacity

* You **do not configure RCUs/WCUs**.
* AWS automatically adjusts capacity as requests come in.
* Best for unpredictable traffic.

So remember:

* **Provisioned Mode** → You can use **DynamoDB Auto Scaling**.
* **On-Demand Mode** → Auto Scaling settings are **not needed**, because AWS already scales the capacity automatically.
