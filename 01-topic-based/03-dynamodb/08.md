8. Question

Category: CSAA – Design High-Performing Architectures

A company currently has an Augment Reality (AR) mobile game that has a serverless backend. It is using a DynamoDB table which was launched using the AWS CLI to store all the user data and information gathered from the players and a Lambda function to pull the data from DynamoDB. The game is being used by millions of users each day to read and store data.

How would you design the application to improve its overall performance and make it more scalable while keeping the costs low? (Select TWO

- `Enable DynamoDB Accelerator (DAX) and ensure that the Auto Scaling is enabled and increase the maximum provisioned read and write capacity.`
- `Use API Gateway in conjunction with Lambda and turn on the caching on frequently accessed data and enable DynamoDB global replication.`
- Since Auto Scaling is enabled by default, the provisioned read and write capacity will adjust automatically. Also enable DynamoDB Accelerator (DAX) to improve the performance from milliseconds to microseconds.
- Configure CloudFront with DynamoDB as the origin; cache frequently accessed data on the client device using ElastiCache.
- Use AWS IAM Identity Center to authenticate users and have them directly access DynamoDB using single sign-on. Manually set the provisioned read and write capacity to a higher RCU and WCU.


This question mixes **4 DynamoDB concepts** that the exam loves to test together. Let's simplify them.

---

# First understand the architecture

```
Mobile App
     |
API Gateway
     |
Lambda
     |
DynamoDB
```

Millions of users are:

* Reading data
* Writing data

So two things matter:

* 🚀 Performance
* 📈 Scalability

---

# Option 1 (✅ Correct)

> Enable DynamoDB Accelerator (DAX) and ensure that Auto Scaling is enabled.

This actually contains **two different optimizations**.

## 1. DAX

Remember:

```
Without DAX

App
 |
Lambda
 |
DynamoDB
```

Every read goes to DynamoDB.

Now:

```
With DAX

App
 |
Lambda
 |
DAX (cache)
 |
DynamoDB
```

Frequently read items come directly from memory.

Result:

```
Milliseconds
      ↓
Microseconds
```

Huge improvement.

---

## 2. Auto Scaling

Suppose today

```
RCU = 100
WCU = 50
```

Suddenly millions of users arrive.

Without Auto Scaling

```
RCU still 100

↓

Throttling
```

With Auto Scaling

```
100
↓

300

↓

700

↓

1500
```

No throttling.

---

Notice what the option says:

> increase the **maximum** provisioned read/write capacity.

Why?

Auto Scaling cannot exceed the maximum you specify.

Example

```
Minimum = 100

Maximum = 1000
```

AWS can scale anywhere between 100 and 1000.

Exactly like EC2 Auto Scaling.

---

# Option 2 (✅ Correct)

> API Gateway caching + Lambda + Global Replication

There are **two ideas** here.

---

## API Gateway Cache

Normally

```
Client

↓

API Gateway

↓

Lambda

↓

DynamoDB
```

Every request reaches Lambda.

Suppose 1 million users ask:

```
Top 10 Players
```

Same request.

API Gateway caches it.

```
Client

↓

API Gateway Cache

↓

Response
```

Lambda is skipped.

Less cost.

Lower latency.

---

## Global Replication

This simply means:

```
USA
 |
DynamoDB

⇅

Europe
 |
DynamoDB
```

People read from the nearest region.

Lower latency.

---

# Why Option 3 is wrong

It says

> Auto Scaling is enabled by default.

❌ Wrong.

The question even tells you

> Table was created using AWS CLI.

For CLI-created provisioned tables, Auto Scaling is **not automatically enabled**.

This is the exam trap.

---

# Why Option 4 is wrong

```
CloudFront

↓

DynamoDB
```

Impossible.

CloudFront cannot use DynamoDB as an origin.

CloudFront works with

* S3
* ALB
* EC2
* API Gateway
* Custom HTTP origins

Not DynamoDB.

Also,

> cache using ElastiCache on client device

❌ ElastiCache runs inside AWS, not on client devices.

---

# Why Option 5 is wrong

IAM Identity Center

↓

Authentication

It has nothing to do with performance.

Also

```
Manual RCU = 10000
Manual WCU = 10000
```

Even at 3 AM...

You're still paying for 10,000.

Auto Scaling is much cheaper.

| Requirement                       | Service                     |
| --------------------------------- | --------------------------- |
| Faster DynamoDB reads             | ✅ DAX                       |
| Scale RCU/WCU automatically       | ✅ DynamoDB Auto Scaling     |
| Trigger Lambda after table change | ✅ DynamoDB Streams          |
| Multi-region replication          | ✅ Global Tables             |
| Cache API responses               | ✅ API Gateway Cache         |
| CDN                               | ✅ CloudFront (not DynamoDB) |

## One important correction

The explanation says **"enable DynamoDB global replication."** Today, the AWS feature name is **DynamoDB Global Tables**. If you see **global replication** in older practice questions, think **Global Tables**.

### A quick way to answer DynamoDB questions

When you see these keywords, immediately think:

* **Slow reads** → **DAX**
* **Throttling due to traffic** → **Auto Scaling** (or increase capacity if appropriate)
* **React to data changes** → **DynamoDB Streams + Lambda**
* **Multi-region / Disaster Recovery** → **Global Tables**
* **Cache API responses** → **API Gateway Cache**
* **Need a CDN** → **CloudFront** (but **not** directly with DynamoDB)

This pattern appears repeatedly in the SAA exam.
