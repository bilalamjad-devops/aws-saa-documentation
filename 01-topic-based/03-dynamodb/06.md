6. Question

Category: CSAA – Design High-Performing Architectures

A Docker application, which is running on an Amazon ECS cluster behind a load balancer, is heavily using Amazon DynamoDB. The application requires improved database performance by distributing the workload evenly and utilizing the provisioned throughput efficiently. Currently, the table’s write capacity units (WCU) are unevenly consumed due to key distribution.

Which of the following should be implemented for the DynamoDB table?

- Use partition keys with low-cardinality attributes, which have a small number of distinct values for each item.
- Reduce the number of partition keys in the DynamoDB table.
- Avoid using a primary key that combines both a partition key and a sort key.
- `Use partition keys with high-cardinality attributes, which have a large number of distinct values for each item.`

This is one of the **most confusing DynamoDB topics**, but once you understand **partitions**, you'll never forget it.

---

# Step 1: What is the problem?

The question says:

> **Write Capacity Units (WCU) are unevenly consumed due to key distribution.**

This sentence tells you everything.

AWS is saying:

> "Some partitions are getting too many writes."

This is called a **hot partition**.

---

# First understand partitions

Imagine DynamoDB is a library.

Instead of one giant shelf, it has many shelves.

```text
DynamoDB

 ├── Partition A
 ├── Partition B
 ├── Partition C
 └── Partition D
```

AWS stores your data across these partitions.

---

# How does DynamoDB decide?

It uses the **Partition Key**.

For example:

```
CustomerID = 101
```

might go to

```
Partition A
```

Another customer:

```
CustomerID = 205
```

goes to

```
Partition C
```

The partition key determines where the item is stored.

---

# Good Example (High Cardinality)

Suppose your partition key is:

```
UserID
```

Data:

```
UserID
1
2
3
4
5
6
100
200
500
```

Thousands or millions of unique values.

AWS spreads them nicely.

```text
Partition A → User 1

Partition B → User 2

Partition C → User 3

Partition D → User 4

...
```

Everyone shares the workload.

Perfect.

---

# Bad Example (Low Cardinality)

Suppose the partition key is:

```
Gender
```

Possible values:

```
Male

Female
```

Only **2 values**.

Now imagine 10 million users.

All males go to one partition.

All females go to another.

```text
Partition A

Male
Male
Male
Male
Male
Male
Male
```

Huge traffic.

Hot partition.

---

Meanwhile:

```text
Partition B

Female
Female
Female
```

Uneven.

---

# Even Worse

Partition key:

```
Country
```

Only:

```
USA

UK

Canada

Pakistan
```

Only 4 values.

Millions of writes.

Only 4 partitions become busy.

Again:

Hot partitions.

---

# What is High Cardinality?

High Cardinality means:

**Lots of unique values.**

Examples:

✅ UserID

✅ CustomerID

✅ OrderID

✅ UUID

Bad examples:

❌ Gender

❌ Country

❌ Status

❌ Department

---

# Why?

Because AWS can distribute data.

Example:

One million users.

Partition key:

```
CustomerID
```

```text
Customer1

Customer2

Customer3

...

Customer1000000
```

AWS spreads them across many partitions.

Balanced.

---

# Question says

> Current write capacity is unevenly consumed.

Translation:

```
One partition is overloaded.
```

Solution?

Increase the number of unique partition key values.

That's exactly what:

> **High Cardinality**

means.

---

# Why not Low Cardinality?

The option says:

> Small number of distinct values.

Example:

```
Status

Pending

Completed
```

Only two values.

All requests hit the same few partitions.

Performance drops.

---

# Why not Reduce Number of Partition Keys?

Think logically.

Current problem:

```
Too few unique values
```

AWS says:

```
Please spread the load.
```

Reducing them makes it even worse.

---

# Why not Avoid Composite Key?

Composite key:

```
Partition Key

+

Sort Key
```

Example:

```
CustomerID

+

OrderDate
```

```
Customer1
2026-01-01

Customer1
2026-02-01

Customer1
2026-03-01
```

This is actually a **good design**.

It lets you efficiently query related items and is commonly used. The performance issue in this question is specifically about **poor partition-key distribution**, not about whether you use a sort key.

---

# Real AWS Example

Bad design:

Partition Key:

```
Country
```

Millions of users.

```
USA

USA

USA

USA

USA
```

One partition becomes extremely busy.

---

Better design:

Partition Key:

```
CustomerID
```

```
1001

1002

1003

1004

1005
```

AWS spreads them automatically.

# Memory Trick

Imagine 1,000 students.

Teacher says:

"Stand in lines according to **Gender**."

Only:

```
Male

Female
```

Huge lines.

---

Instead:

"Stand according to **Student ID**."

```
1

2

3

4

5

6
```

Everyone spreads evenly.

Exactly how DynamoDB likes it.

---

# Golden Rule #24

Whenever the exam says:

* **Hot partition**
* **Uneven write capacity**
* **Uneven read capacity**
* **Partition key problem**
* **Improve throughput**

Think:

✅ **High Cardinality Partition Key**

## Quick Revision Table

| Term                      | Meaning                                         | Good or Bad?                                                                  |
| ------------------------- | ----------------------------------------------- | ----------------------------------------------------------------------------- |
| **Partition Key**         | Determines where data is stored                 | ✅ Important                                                                   |
| **High Cardinality**      | Many unique values (UserID, OrderID)            | ✅ Best                                                                        |
| **Low Cardinality**       | Few unique values (Gender, Country, Status)     | ❌ Can create hot partitions                                                   |
| **Hot Partition**         | One partition receives disproportionate traffic | ❌ Avoid                                                                       |
| **Composite Primary Key** | Partition Key + Sort Key                        | ✅ Useful when your access patterns require grouping and sorting related items |
