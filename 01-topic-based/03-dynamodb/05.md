5. Question

Category: CSAA – Design Secure Architectures

A GraphQL API hosted is hosted in an Amazon EKS cluster with AWS Fargate launch type and deployed using AWS SAM. The API is connected to an Amazon DynamoDB table with DynamoDB Accelerator (DAX) as its data store. Both resources are hosted in the us-east-1 region.

The AWS IAM authenticator for Kubernetes is integrated into the EKS cluster for role-based access control (RBAC) and cluster authentication. A solutions architect must improve network security by preventing database calls from traversing the public internet. An automated cross-account backup for the DynamoDB table is also required for long-term retention.

Which of the following should the solutions architect implement to meet the requirement?


- Create a DynamoDB gateway endpoint. Set up a Network Access Control List (NACL) rule that allows outbound traffic to the dynamodb.us-east-1.amazonaws.com gateway endpoint. Use the built-in on-demand DynamoDB backups for cross-account backup and recovery.

- Create a DynamoDB interface endpoint. Set up a stateless rule using AWS Network Firewall to control all outbound traffic to only use the dynamodb.us-east-1.amazonaws.com endpoint. Integrate the DynamoDB table with Amazon Timestream to allow point-in-time recovery from a different AWS account.
- Create a DynamoDB interface endpoint. Associate the endpoint to the appropriate route table. Enable Point-in-Time Recovery (PITR) to restore the DynamoDB table to a particular point in time on the same or a different AWS account.
- `Create a DynamoDB gateway endpoint. Associate the endpoint to the appropriate route table. Use AWS Backup to automatically copy the on-demand DynamoDB backups to another AWS account for disaster recovery.`

This is a **classic Tutorials Dojo "overloaded" question**. They intentionally throw in many AWS services to distract you.

Let's learn how to solve it like an AWS Solutions Architect.

# Requirement 1

> Prevent database calls from traversing the public internet.

Question:

How does an EC2/EKS application privately access DynamoDB?

Immediately think:

**VPC Endpoint**

Now ask yourself:

Is DynamoDB a **Gateway Endpoint** or **Interface Endpoint** service?

This is a very important SAA memory point.

| Service                                                   | Endpoint Type      |
| --------------------------------------------------------- | ------------------ |
| **S3**                                                    | Gateway Endpoint   |
| **DynamoDB**                                              | Gateway Endpoint   |
| Most other AWS services (SQS, SNS, Secrets Manager, etc.) | Interface Endpoint |

👉 **Memorize this.**

So the answer is:

**Gateway Endpoint.**

---

## How does it work?

Without a VPC endpoint:

```text
EKS
   │
Internet Gateway
   │
Public DynamoDB Endpoint
```

Traffic stays on the AWS network, but it still uses the **public service endpoint**.

---

With a Gateway Endpoint:

```text
EKS
   │
Gateway Endpoint
   │
DynamoDB
```

No Internet Gateway is required for DynamoDB traffic.

Everything stays private within AWS.

---

# Why Route Table?

A Gateway Endpoint works by adding routes.

When you create it:

AWS automatically adds a route like:

```
Destination:
DynamoDB Prefix List

Target:
Gateway Endpoint
```

That's why the answer says:

> Associate the endpoint to the appropriate route table.

---

# Requirement 2

> Cross-account backup

Question:

Can DynamoDB's built-in backup copy backups to another AWS account?

Answer:

❌ No.

Built-in backup only creates backups **inside the same account**.

---

So what service supports:

* Cross-account
* Cross-region
* Central backup management
* Long-term retention

Answer:

✅ AWS Backup

---

Architecture becomes:

```text
Application

↓

Gateway Endpoint

↓

DynamoDB

↓

AWS Backup

↓

Another AWS Account
```

---

# Why not PITR?

PITR = Point-in-Time Recovery.

Think of PITR as:

"I accidentally deleted data at 3:15 PM."

Restore to:

```
3:14 PM
```

Great.

But:

Can PITR copy data to another account?

❌ No.

PITR is **recovery**, not **backup management**.

---

# Why not Interface Endpoint?

Very common exam trap.

Remember this table:

| AWS Service     | Endpoint  |
| --------------- | --------- |
| S3              | Gateway   |
| DynamoDB        | Gateway   |
| Everything else | Interface |

Whenever you see DynamoDB or S3:

👉 Think **Gateway Endpoint**.

## Tiny correction to keep your AWS knowledge future-proof

The explanation says DynamoDB traffic "uses the Internet Gateway." Conceptually, the exam wants you to choose a **Gateway VPC Endpoint**, but in reality, AWS service traffic to public endpoints stays on the AWS global network and doesn't literally traverse the public Internet. A **Gateway VPC Endpoint** ensures the traffic uses private routing within your VPC and avoids using public service endpoints. For the SAA exam, the important takeaway is still:

* **Private access to S3/DynamoDB → Gateway VPC Endpoint**
* **Cross-account DynamoDB backups → AWS Backup**
