4. Question

Category: CSAA – Design High-Performing Architectures

A leading IT consulting company has an application which processes a large stream of financial data by an Amazon ECS Cluster then stores the result to a DynamoDB table. You have to design a solution to detect new entries in the DynamoDB table then automatically trigger a Lambda function to run some tests to verify the processed data.

What solution can be easily implemented to alert the Lambda function of new entries while requiring minimal configuration change to your architecture?

- `Enable DynamoDB Streams to capture table activity and automatically trigger the Lambda function.`
- Use CloudWatch Alarms to trigger the Lambda function whenever a new entry is created in the DynamoDB table.
- Invoke the Lambda functions using SNS each time that the ECS Cluster successfully processed financial data.
- Use Systems Manager Automation to detect new entries in the DynamoDB table then automatically invoke the Lambda function for processing.

You're already seeing the pattern now. This question is actually much easier than the previous one.

---

# Step 1: Ignore the story

Forget:

* IT consulting
* Financial data
* ECS

The only important sentence is:

> **Detect new entries in the DynamoDB table and automatically trigger a Lambda function.**

That's the whole question.

---

# Step 2: Draw the architecture

Current architecture:

```text
ECS
   │
   ▼
DynamoDB
```

Now they want:

> Whenever a **new item** is inserted into DynamoDB...

...automatically run Lambda.

So it becomes:

```text
ECS
   │
   ▼
DynamoDB
   │
   ▼
DynamoDB Streams
   │
   ▼
Lambda
```

Done.

---

# Why DynamoDB Streams?

Remember what we learned in the previous question:

DynamoDB Streams is used when something **already happened** in DynamoDB.

Examples:

* New item inserted ✅
* Item updated ✅
* Item deleted ✅

Whenever one of these occurs:

```text
DynamoDB

↓

Streams

↓

Lambda
```

---

# The key phrase

The question says:

> Detect **new entries** in DynamoDB.

That immediately tells you:

A change happened **inside** DynamoDB.

Which AWS service watches for changes?

✅ DynamoDB Streams.

---

# Why not CloudWatch?

Many students get confused here.

CloudWatch monitors **metrics**, not **data**.

It knows things like:

* CPU = 80%
* Memory (custom metric)
* Request Count
* Network In

It **does not know**:

```text
New row inserted

User updated

Customer deleted
```

Those are database events, not CloudWatch metrics.

---

# Why not SNS?

Think about the sequence.

The option says:

```text
ECS

↓

SNS

↓

Lambda
```

The problem is:

The question asks:

> Trigger Lambda **after a new entry appears in DynamoDB**.

SNS doesn't know if DynamoDB actually saved the item successfully.

Maybe:

```text
ECS

↓

SNS

↓

Lambda

↓

Oops...

DynamoDB write failed
```

Now Lambda runs even though the record doesn't exist.

That's not what we want.

Instead:

```text
ECS

↓

DynamoDB

↓

Record successfully written

↓

Streams

↓

Lambda
```

Now Lambda runs **only after the database changes**.

Much safer.

---

# Why not Systems Manager?

Systems Manager is for managing infrastructure.

Examples:

* Patch EC2
* Run shell commands
* Automation documents
* Session Manager

It has nothing to do with monitoring DynamoDB table changes.

---

# Exam Shortcut

Whenever you read:

* New record inserted
* Item updated
* Item deleted
* React to database changes
* Automatically invoke Lambda after database update

Don't even read the options first.

Immediately think:

> **DynamoDB Streams → Lambda**
