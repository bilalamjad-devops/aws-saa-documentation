3. Question

Category: CSAA – Design High-Performing Architectures

A healthcare organization wants to build a system that can predict drug prescription abuse. The organization will gather real-time data from multiple sources, which include Personally Identifiable Information (PII). It’s crucial that this sensitive information is anonymized prior to landing in a NoSQL database for further processing.

Which solution would meet the requirements?



- Stream the data in an Amazon DynamoDB table. Enable DynamoDB Streams, and configure an AWS Lambda function with AmazonDynamoDBFullAccess permissions to perform anonymization on newly written items.
- Create a data lake in Amazon S3 and use it as the primary storage for patient health data. Use an S3 trigger to run an AWS Lambda function that performs anonymization. Send the anonymized data to Amazon DynamoDB.
- Deploy an Amazon Data Firehose stream to capture and transform the streaming data. Deliver the anonymized data to Amazon Redshift for analysis.
- `Ingest real-time data using Amazon Kinesis Data Stream. Use an AWS Lambda function to anonymize the PII, then store it in Amazon DynamoDB.`




This is an **excellent SAA question** because it teaches a very important principle:

> **Where should you process the data?**
>
> **Before storage or after storage?**

This is what the entire question is testing.

# Step 2: What is PII?

PII = Personally Identifiable Information.

Example:

```text
Name

CNIC

Passport

Email

Phone Number
```

Suppose incoming data is:

```text
Name: Bilal Amjad
Age: 26
Disease: Diabetes
```

Requirement:

Don't store:

```text
Bilal Amjad
```

Store something like:

```text
Patient-001
Age:26
Disease:Diabetes
```

This is called **anonymization**.

---

# Step 3: The important sentence

The question says:

> **before landing in a NoSQL database**

Read it carefully.

It DOES NOT say:

> After storing...

It says:

> Before storing.

That changes everything.

---

# Wrong Architecture

```text
Data

↓

DynamoDB

↓

Streams

↓

Lambda

↓

Anonymize
```

Why wrong?

Because sensitive data has **already been stored**.

Too late.

---

# Correct Architecture

```text
Data

↓

Kinesis Data Streams

↓

Lambda

↓

Anonymize

↓

DynamoDB
```

Now PII never reaches DynamoDB.

Perfect.

---

# Why Kinesis?

Think of Kinesis as a **conveyor belt**.

Imagine an airport security checkpoint.

Passengers arrive.

```text
Passenger

↓

Security Check

↓

Airport
```

Security checks them **before** entering.

Kinesis does the same.

Incoming data arrives.

Before storing,

Lambda processes it.

---

# What does Lambda do?

Incoming:

```text
Name = Bilal

Email = bilal@gmail.com
```

Lambda changes it to:

```text
Name = Patient-928

Email = Hidden
```

Now it stores:

```text
Patient-928
```

No sensitive data.

---

# Why not DynamoDB Streams?

Let's use the same example.

Incoming:

```text
Bilal
```

Immediately stored:

```text
DynamoDB
```

Only then:

```text
Streams

↓

Lambda

↓

Hide Bilal
```

Too late.

The original data already existed in DynamoDB.

Requirement violated.

---

# Why not S3?

Same issue.

Data first goes into S3.

Only afterward does Lambda anonymize it.

Again,

PII was stored.

Wrong.

---

# Why not Firehose + Redshift?

This is a trick.

Firehose can transform data.

That's okay.

But where does it store?

Redshift.

Question asked for:

> **NoSQL database**

Redshift is:

❌ Relational Data Warehouse

Not NoSQL.

Wrong answer.

# Comparison (Very Important)

### DynamoDB Streams

```text
DynamoDB

↓

Streams

↓

Lambda
```

Used when:

> Something **already stored** changes.

Keywords:

* update
* insert
* delete
* notification
* audit

---

### Kinesis Data Streams

```text
Incoming Data

↓

Kinesis

↓

Lambda

↓

Store
```

Used when:

> Incoming data must be processed **before** storage.

Keywords:

* real-time
* streaming
* IoT
* clickstream
* sensors
* logs
* financial transactions

---

# Golden Rule #22

There are two different "Streams" in AWS:

### **Kinesis Data Streams**

👉 Before storage

Processes live incoming data.

---

### **DynamoDB Streams**

👉 After storage

Processes changes that already happened in DynamoDB.

---

## A memory trick you'll never forget

Imagine a school.

### Kinesis = School Gate 🚪

Students are checked **before entering** the school.

```
Student
   ↓
Gate Check
   ↓
School
```

---

### DynamoDB Streams = CCTV 📹

Students are already inside the school.

CCTV records what they do.

```
School
   ↓
CCTV records changes
```

---

This one picture helps answer many SAA questions:

* **Need to transform/filter/anonymize data before saving?** → **Kinesis + Lambda**
* **Need to react to inserts/updates/deletes after they're saved?** → **DynamoDB Streams + Lambda**
